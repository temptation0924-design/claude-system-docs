# Marketing Design Planner

마케팅 목적의 디자인을 빠르게 구조화하고 실행 패키지로 전환하는 **기획 파트너**.
디자인을 직접 그리는 AI가 아니라, 디자인을 쉽게 뽑을 수 있도록 사고를 구조화하는 엔진.

## When to Use

- 문서, 랜딩페이지, 블로그 이미지 등 마케팅 결과물의 **방향을 잡을 때**
- "이 주제로 자료 만들어야 하는데" — 뭘 먼저 정해야 할지 모를 때
- 카피, 구조, 비주얼 컨셉을 **한 번에 패키지로** 받고 싶을 때
- 참고자료는 있지만 어떻게 풀어야 할지 모를 때
- 기존 초안이 약해서 **어디를 살리고 바꿔야 할지** 알고 싶을 때
- 디자이너/카피라이터에게 **바로 전달할 브리프**가 필요할 때

**When NOT to use:**
- 이미 디자인 방향이 확정되고 실행만 남은 경우 → `/frontend-design`, `/supanova-design-engine`
- 브랜드 디자인 시스템 전체 구축 → `/design-consultation`
- 기존 사이트 디자인 감사 → `/design-review`

## Core Principle

```
질문은 최소로, 패키지는 최대로.
한 줄 주제만으로도 실행 가능한 기획 패키지를 만든다.
```

## Phase 0: Routing — 워크플로우 분기

사용자 입력을 받으면 **먼저 어떤 경로인지 판단**한다.

### 0A. 워크플로우 선택

| 입력 신호 | 경로 | 설명 |
|-----------|------|------|
| "~기획해줘", "~만들어줘", 주제만 제시 | **신규 기획** | Phase 1부터 순서대로 |
| "이 초안 봐줘", "약한데", "고쳐줘", 기존 자료 첨부 | **수정 기획** | Phase R (Revision) 진입 |
| "브리프 만들어줘", "디자이너한테 전달", "바로 쓸 수 있게" | **핸드오프** | Phase H (Handoff) 진입 |
| 불분명 | 신규 기획 기본 | 질문 1개로 확인 |

### 0B. 복잡도 게이트

| 복잡도 | 조건 | 질문 수 | 속도 |
|--------|------|---------|------|
| **Fast** | 단일 채널, 단순 요청 | 2~4개 | ~3분 |
| **Standard** | 1~2채널, 일반 상업 요청 | 3~5개 | ~5분 |
| **Extended** | 멀티채널, 규제 도메인, 다수 이해관계자 | 5~7개 | ~8분 |

복잡도는 자동 판단한다. Extended로 판단되면 "질문이 조금 더 필요합니다"라고 먼저 안내.

**3~5개 규칙은 Fast/Standard 기본값이다. Extended에서는 넘을 수 있다. 단, 8개 초과는 금지.**

## Phase 1: Input & Detection (신규 기획)

사용자 입력을 받으면 **결과물 유형을 자동 감지**한다.

| 입력 신호 | 감지 유형 | 패키지 구성 |
|-----------|----------|------------|
| "제안서", "보고서", "소개서" | 문서 | 목차 + 섹션별 카피 + 핵심 문구 + 표지 이미지 컨셉 |
| "랜딩페이지", "홈페이지", "세일즈페이지", "상세페이지" | 랜딩페이지 | 히어로~CTA 섹션 구조 + 카피 + 비주얼 방향 |
| "블로그 이미지", "썸네일", "배너" | 블로그 이미지 | 제목 타이포 + 인라인 문구 + 비주얼 컨셉 + 생성 프롬프트 |
| "카드뉴스", "SNS 콘텐츠", "인스타" | SNS | 슬라이드별 구조 + 카피 + 비주얼 톤 |
| 2개 이상 채널 동시 언급 | 멀티채널 | 공통 코어 + 채널별 적응 패키지 |
| 복합 또는 불분명 | 복합 | 사용자에게 확인 후 조합 |

## Phase 2: Selective Questioning

### Essentials 체크 (6가지)

질문 전에 **이미 파악된 정보를 먼저 확인**한다.

| Essential | 왜 중요한가 | 이미 있으면 |
|-----------|------------|-----------|
| 채널 | 구조와 정보 밀도 결정 | 스킵 |
| 톤 | 카피와 비주얼 방향 결정 | 스킵 |
| 목적/전환 | 메시지 우선순위 결정 | 스킵 |
| 타겟 | 약속과 반론 대응 결정 | 스킵 |
| 주제/오퍼 | 자료의 본질 결정 | 스킵 |
| CTA 의도 | 전환 행동 결정 | 스킵 |

**3개 이상 이미 파악되면 → 바로 드래프트, 확인만 받는다.**

### 참고자료 적응 규칙 (G1)

입력에 포함된 정보량에 따라 질문 수를 **자동 축소**한다.

| 입력 상태 | 질문 전략 | 예시 |
|-----------|----------|------|
| 주제 한 줄만 | 복잡도 게이트 기준 질문 | "부동산 세미나 홍보" → 채널, 톤, 타겟 순서로 |
| 주제 + 채널/타겟 명시 | 1~2개 확인 질문 | "50대 건물주 대상 랜딩페이지" → 톤만 확인 |
| 주제 + 참고자료 충분 | 0개 → 즉시 요약 확인 | "이 자료 기반으로 만들어줘" → "이렇게 이해했는데 맞나요?" |

**핵심: 이미 알려준 정보를 다시 묻지 않는다.**

### 질문 우선순위

| 순위 | 질문 영역 | 예시 | 항상 묻는가? |
|------|----------|------|------------|
| 1 | 채널/매체 | "어디에 쓸 건가요? (문서/웹/SNS/인쇄)" | YES |
| 2 | 톤/키메시지 | "어떤 느낌이면 좋겠어요? (신뢰감/트렌디/따뜻함/전문적)" | YES |
| 3 | 타겟 | "누가 보는 자료인가요?" | 불분명할 때만 |
| 4 | 목적/전환 | "이걸 보고 뭘 하길 바라나요?" | 랜딩페이지일 때 |
| 5 | 제약/선호 | "반드시 들어가야 하는 내용이나 피할 점?" | 참고자료 없을 때 |
| 6 | 차별화 | "경쟁사 대비 핵심 차별점?" | Extended일 때 |
| 7 | 증거 | "보여줄 수 있는 숫자/후기/사례?" | Extended일 때 |

### 질문 형식 규칙

```
1. 한 번에 하나의 질문만 던진다
2. 가능하면 2~4개 선택지를 제시한다 (+ "직접 입력" 옵션)
3. 사용자가 "알아서 해줘"라고 하면 → 합리적 기본값으로 진행하고 결과 제시
4. 참고자료가 채널+톤+타겟을 이미 포함하면 → 질문 스킵, 확인만 받는다
5. 절대 질문을 한꺼번에 묶어 던지지 않는다 (Extended에서도 하나씩)
```

## Phase 3: Engine Modules

질문 확인 후, 결과물 유형에 따라 엔진 모듈을 **자동 조합**한다.

**필수 모듈 규칙 (G2): 4개 엔진 모두 항상 실행한다.** 비주얼 엔진은 선택이 아닌 필수 — 문서든 랜딩이든 블로그든 모든 패키지에 비주얼 가이드(무드+컬러+프롬프트)가 포함되어야 한다. "이미지 관련 요청이 아니니까 비주얼은 스킵"은 허용하지 않는다.

**예외: 수정 경로(Phase R)에서는 약한 엔진만 부분 실행 가능.**

### 3A. Message Engine (메시지 엔진)

StoryBrand 프레임워크 기반 — 고객을 주인공으로, 브랜드를 가이드로 포지셔닝.

**필수 출력물 (G5 — 원라이너는 반드시 생성):**
- **핵심 약속 (one-liner)**: "고객이 [문제]를 겪을 때, [우리 방식]으로 [결과]를 얻게 합니다" — **모든 패키지의 첫 번째 항목. 생략 금지.**
- 헤드라인 3종 (직접적 / 감성적 / 질문형)
- CTA 문구 2종
- Objection 대응 카피 (주요 반론 2~3개 + 해소 문구)

> 원라이너가 없으면 나머지 카피가 방향 없이 흩어진다. 원라이너 → 헤드라인 → CTA 순서로 파생.

### 3B. Structure Engine (구조 엔진)

CRO(전환율 최적화) 관점에서 섹션 순서와 전환 포인트를 설계.

**문서용 구조:**
| 섹션 | 역할 |
|------|------|
| 표지/인트로 | 핵심 약속 전달 |
| 문제 정의 | 공감 + 긴급성 |
| 해결 방안 | 우리만의 접근 |
| 증거/사례 | 신뢰 구축 |
| 제안/CTA | 다음 행동 유도 |

**랜딩페이지용 구조:**
| 섹션 | 역할 | 전환 요소 |
|------|------|----------|
| Hero | 핵심 가치 + CTA | 주 CTA 버튼 |
| Problem | 고객 고통 공감 | — |
| Solution | 우리 방식 제시 | — |
| Social Proof | 후기/숫자/로고 | 보조 CTA |
| How It Works | 3단계 프로세스 | — |
| Objection | 반론 해소 | FAQ 형태 |
| Final CTA | 마지막 전환 | 주 CTA 반복 |

### 3C. Content Engine (콘텐츠 엔진)

**출력물:**
- 아웃라인 (섹션별 핵심 메시지 1줄)
- 본문 초안 (섹션별 2~3문장 스켈레톤)
- 톤 가이드 (문체, 호칭, 금지 표현)

### 3D. Visual Engine (비주얼 엔진)

**톤 → 비주얼 자동 매핑:**

| 톤 | 컬러 방향 | 레이아웃 | 이미지 톤 | 서체 계열 |
|----|----------|---------|----------|----------|
| 신뢰·전문 | 네이비/그레이/화이트 | 미니멀, 여백 많음 | 실제 사진, 깔끔 | 고딕 정자체 (Pretendard) |
| 따뜻·친근 | 웜톤, 베이지/코랄 | 카드형, 둥근 모서리 | 일러스트 또는 따뜻한 사진 | 둥근 고딕 (Noto Sans KR) |
| 모던·세련 | 다크/모노톤 + 포인트 | 대비형, 비대칭 | 추상적, 그래픽 | 산세리프 (Outfit, Geist) |
| 강렬·임팩트 | 레드/옐로우 + 블랙 | 대형 타이포, 풀블리드 | 강한 대비 사진 | 볼드 고딕 (Black weight) |

Phase 2에서 확정된 톤을 기반으로 위 매핑을 자동 적용한다. 사용자가 별도 지정하면 오버라이드.

**출력물:**
- 무드 키워드 3~5개 (예: "clean, warm, professional, light")
- 컬러 팔레트 제안 (primary + secondary + accent, HEX 코드)
- 타이포그래피 방향 (위 매핑 기반 서체 + 크기 위계)
- 이미지 생성 프롬프트 (결과물별 1~3개, DALL-E/Midjourney 호환)

## Phase 4: Package Assembly

엔진 출력을 **결과물 유형에 맞는 패키지**로 조립한다.

```

### 3. 섹션별 카피 초안
#### 섹션 1: ...

### 4. 비주얼 가이드

- 컬러: ...
- 표지 이미지 프롬프트: ...

```
## 기획 패키지: [제목]

- 무드: ...
- 레이아웃: ...

### 2. 슬라이드별 카피

### 4. 이미지 생성 프롬프트

- 본문 배경/아이콘: ...

## Phase R: Revision Workflow (수정 경로)

기존 초안/자료가 있을 때 진입하는 경로. **처음부터 재기획하지 않는다.**

### R1. 5가지 진단

| 진단 영역 | 확인 사항 | 판정 |
|-----------|----------|------|
| 목적 적합성 | 이 초안이 실제 비즈니스 목표를 지원하는가? | OK / 약함 / 불일치 |
| 타겟 적합성 | 맞는 사람에게 말하고 있는가? | OK / 약함 / 불일치 |
| 채널 적합성 | 구조가 사용 환경에 맞는가? | OK / 약함 / 불일치 |
| 톤 적합성 | 목소리가 약하거나 브랜드에서 벗어나지 않는가? | OK / 약함 / 불일치 |
| 전환 논리 | CTA와 반론 대응이 있는가? | OK / 약함 / 없음 |

### R2. 3분류 — 유지 / 수정 / 방향전환

진단 결과를 반드시 3가지 버킷으로 분류해서 제시한다.

| 분류 | 의미 | 예시 |
|------|------|------|
| **유지** | 잘 되어 있음, 건드리지 않는다 | "5섹션 구조 좋음", "비용 투명 공개 유지" |
| **수정** | 방향은 맞지만 더 강하게 | "사례에 구체적 숫자 추가", "헤드라인 차별화" |
| **방향전환** | 근본적으로 접근을 바꿔야 | "기능 나열 → 고통 해결형 리프레이밍" |

### R3. 엔진 부분 실행

진단에서 "약함" 또는 "불일치"인 영역의 엔진만 실행한다.

| 진단 결과 | 실행할 엔진 |
|-----------|-----------|
| 메시지 약함 | 메시지 엔진만 |
| 구조 약함 | 구조 엔진만 |
| 전체적으로 약함 | 4엔진 전부 (G2 적용) |
| 구조 OK + 메시지 약함 | 메시지 + 콘텐츠만 |

### R4. 수정 패키지 출력

```
## 수정 패키지: [제목]

### 1. 진단 요약
| 영역 | 판정 | 핵심 이슈 |
|------|------|----------|

### 2. 유지할 것
- ...

### 3. 수정할 것
| 위치 | 현재 | 수정 방향 |
|------|------|----------|

### 4. 방향전환
| 위치 | 현재 접근 | 새 접근 | 이유 |
|------|----------|--------|------|

### 5. 강화된 메시지 (해당 시)
- One-liner: ...
- 변경 전 vs 변경 후: ...

### 6. 실행 연결
- ...
```

## Phase 5: Quality Gate

패키지 완성 후 **자동으로 8항목을 점검**한다. 사용자에게 보여주지 않되, 누락 항목이 있으면 보완 후 전달.

| # | 체크포인트 | 기준 |
|---|----------|------|
| 1 | 목적 명확성 | 비즈니스/커뮤니케이션 목표가 명확한가 |
| 2 | 타겟 적합성 | 메시지가 올바른 타겟의 고통에 맞는가 |
| 3 | 채널 적합성 | 구조와 밀도가 실제 매체에 맞는가 |
| 4 | 톤 일관성 | 목소리와 무드가 전체적으로 일관되는가 |
| 5 | 증거 논리 | 약속을 뒷받침하는 증거가 있는가 |
| 6 | 전환 논리 | CTA와 반론 대응이 빠지지 않았는가 |
| 7 | 핸드오프 준비 | 협업자가 바로 사용할 수 있는 수준인가 |
| 8 | 정보 규율 | 불필요한 부풀림 없이 실용적인가 |

**누락 발견 시**: 자동 보완 후 전달. "Quality Gate에서 [X]가 약해서 보강했습니다" 한 줄 안내.

## Phase E: Execution Bridge — 3-Stage 디자인 파이프라인

패키지 완성 후 **"만들어줘"**가 트리거. 3단계로 기획을 완성된 디자인으로 변환한다.

**핵심 철학: Midjourney가 방향을 보여주고 → Figma가 구조를 잡고 → Code가 완성한다.**

### 실행 경로 자동 선택

|------------|----------|------|
| 블로그 이미지 / 썸네일 / 배너 | **Stage 1→2→3 풀코스** | 메인 워크플로우 |
| SNS / 카드뉴스 | **Stage 1→2→3** (Stage 1은 그래픽 트랙) | 사진보다 타이포 중심 |
| 랜딩페이지 | `/frontend-design` → `/supanova-design-engine` | 별도 스킬 연결 |
| 랜딩페이지 배포 | `/landing-page-deploy` | 별도 스킬 연결 |
| 문서 (PPT) | `/pptx` | 별도 스킬 연결 |
| 문서 (PDF) | `/pdf` | 별도 스킬 연결 |
| 프레젠테이션 | `/frontend-slides` | 별도 스킬 연결 |
| 디자인 시스템 | `/design-consultation` | 별도 스킬 연결 |

### Stage 1: Midjourney — 시각적 방향 탐색

**목적: "어떤 느낌인지" 눈으로 보고 결정한다. 정답을 만드는 게 아니라 방향을 찾는 것.**

빈 캔버스에서 시작하면 막막하다. Midjourney가 여러 방향을 보여주면 "이거!" 하고 고를 수 있다.

#### Stage 1에서 결정하는 것

| 결정 항목 | 예시 |
|-----------|------|
| **색감 방향** | "이 이미지의 네이비+골드 조합이 좋아" |
| **구도/레이아웃 감** | "좌우 분할이 좋아" or "상단 텍스트+하단 이미지" |
| **무드/분위기** | "이 어두운 톤이 맞아" or "좀 더 밝게" |
| **배경 이미지** | 마음에 드는 이미지 자체를 배경으로 활용 |

#### 3트랙 동시 탐색 — 항상 3가지 완전히 다른 방향을 보여준다

| 트랙 | 방향 | 프롬프트 스타일 | 용도 |
|------|------|---------------|------|
| A. 포토 리얼 | 실제 사진 느낌 | `editorial photography, cinematic` | 사진 배경이 필요할 때 |
| B. 그래픽/인포 | 추상적, 데이터 시각화 | `infographic, minimal, geometric` | 숫자/데이터 중심일 때 |
| C. 컨셉/일러스트 | 감성적, 아트 방향 | `illustration, conceptual, metaphor` | 감성/스토리 중심일 때 |

**각 트랙 4장씩 → 12장 중에서 "이 느낌!" 선택.**

#### 프롬프트 구조 규칙

```
[주제 설명], [스타일/무드], [구도/앵글], [조명], [컬러 톤] --ar [비율] --v 6.1 --s [값] --no text, watermark, logo, words, letters, subtitle
```

**종횡비 자동 매칭:**

| 용도 | 비율 | Midjourney 파라미터 |
|------|------|-------------------|
| 네이버 블로그 썸네일 | 16:9 | `--ar 16:9` |
| 티스토리 대표이미지 | 16:9 | `--ar 16:9` |
| 인스타그램 피드 | 1:1 | `--ar 1:1` |
| 인스타그램 스토리 | 9:16 | `--ar 9:16` |
| 카드뉴스 | 1:1 또는 4:5 | `--ar 1:1` 또는 `--ar 4:5` |
| 유튜브 썸네일 | 16:9 | `--ar 16:9` |

**스타일라이즈 자동 선택 (톤 매핑 연동):**

| 톤 | `--s` 값 | 이유 |
|----|---------|------|
| 신뢰·전문 | `--s 250` | 과한 스타일링 억제, 깔끔함 |
| 따뜻·친근 | `--s 500` | 적당한 감성 표현 |
| 모던·세련 | `--s 750` | 아트 디렉션 강조 |
| 강렬·임팩트 | `--s 750` | 극적 표현 허용 |

**네거티브 프롬프트 필수**: `--no text, watermark, logo, words, letters, subtitle` — 텍스트는 Stage 2~3에서 입히므로 이미지에 텍스트가 들어가면 안 된다.

#### 변형/반복 대응

| 대표님 피드백 | Claude 대응 |
|-------------|-----------|
| "3번이 좋아" | 해당 프롬프트 기반 `--seed` 미세 변형 3종 추가 |
| "더 밝게 / 더 어둡게" | 조명과 컬러 톤 파라미터 조정 |
| "다른 느낌으로" | 구도와 스타일 변경, 무드 유지 |
| "A트랙이랑 B트랙 섞어줘" | 두 트랙의 요소를 조합한 하이브리드 프롬프트 |
| "이 이미지 기반으로 변형" | `--iw` 이미지 가중치 + 텍스트 보정 |

#### Stage 1 → Stage 2 핸드오프

```
[Stage 1 산출물]
- 선택한 Midjourney 이미지 (1~2장)
- 확정: 색감 방향, 구도 감, 무드, 배경 이미지
- 미확정: 텍스트 위치, 정확한 비율, 서체 크기
→ 대표님: "이 느낌으로 진행해" or "이 이미지로 가자"
```

### Stage 2: Figma — 레이아웃 확정

**목적: Stage 1의 "느낌"을 "구조"로 고정한다. 여기서 레이아웃이 확정된다.**

Midjourney 이미지를 배경에 깔고, 위에 텍스트를 배치해보면서 구조를 잡는다. 드래그로 빠르게 위치/크기를 실험할 수 있다.

#### Stage 2에서 결정하는 것

| 결정 항목 | 예시 |
|-----------|------|
| **텍스트 위치** | 상단 좌측 정렬 / 중앙 / 하단 오버레이 |
| **정보 위계** | 제목 > 서브 > 숫자 > 뱃지 순서와 크기 비율 |
| **여백과 비율** | 이미지 영역 60% + 텍스트 영역 40% |
| **서체 + 크기 확정** | Pretendard Black 48px / Regular 16px |
| **오버레이 농도** | 이미지 위 반투명 레이어 투명도 |
| **전체 구성 밸런스** | 요소 간 간격, 시선 흐름 |

#### Figma 작업 방식

Claude가 MCP로 초기 레이아웃을 자동 생성한 후, 대표님이 조정한다.

**Claude 자동 생성 (Figma MCP):**

| 요소 | 자동 생성 내용 |
|------|-------------|
| 프레임 | 채널 사이즈 프리셋 (1200x630 등) |
| 배경 레이어 | Stage 1 이미지 삽입 가이드 (점선 테두리) 또는 컬러 배경 |
| 메인 카피 | 패키지의 헤드라인 — 위치/크기/서체/컬러 적용 |
| 서브 카피 | 서브 텍스트 배치 |
| 데이터 블록 | 핵심 숫자 있으면 하단에 배치 |
| 컬러 스타일 | 패키지 팔레트를 Figma 스타일로 등록 |
| 오버레이 레이어 | 이미지 위 반투명 그라데이션 |

**⚠️ Figma MCP 한글 폰트 규칙 (실전 검증됨):**
- **Inter 사용 금지** — 한글 미지원, 글자가 깨짐
- **Noto Sans KR 사용** — Black/Bold/Medium/Regular 4종 로드
- 폰트 로드 실패 시 → Noto Sans CJK KR 폴백 시도
- 강조 텍스트(accent)는 별도 텍스트 노드로 분리 (Figma에서 부분 색상 변경 어려움)

**⚠️ 색감 추출 워크플로우 (Stage 1 이미지 → 팔레트):**
- 대표님이 Midjourney/참고 이미지를 공유하면 → 색감 6종 자동 추출 (배경/메인/강조/보조/포인트/글로우)
- 추출한 팔레트를 Figma + HTML 양쪽에 동일 적용
- 밝은 배경 선택 시 → 텍스트 색상 자동 반전 (화이트→다크)

**대표님 조정:**
```
1. Midjourney 이미지를 배경 레이어에 드래그
2. 텍스트 위치/크기를 드래그로 조정
3. 오버레이 투명도를 눈으로 보면서 조절
4. "이 구조로 확정" → Figma URL 공유
```

#### Stage 2에서 하지 않는 것 (Stage 3에서 할 것)

| Figma에서 하기 어려운 것 | Stage 3에서 해결 |
|------------------------|----------------|
| 한글 자간 미세 조정 (-0.02em) | CSS `letter-spacing` |
| 텍스트 컬러 그라데이션 | CSS `background-clip: text` |
| 복잡한 오버레이 블렌딩 | CSS `mix-blend-mode` |
| 정확한 px 단위 정렬 | HTML/CSS 직접 제어 |
| 배경 블러 효과 | CSS `backdrop-filter` |
| 텍스트 그림자 정밀 제어 | CSS `text-shadow` 다중 레이어 |

#### Stage 2 → Stage 3 핸드오프

```
[Stage 2 산출물]
- Figma URL (또는 스크린샷)
- 확정: 텍스트 위치, 정보 위계, 이미지+텍스트 비율, 서체, 컬러
- 미확정: 미세 간격, 효과, 최종 퀄리티 디테일
→ 대표님: "코드로 다듬어줘" or Figma URL 공유
```

### Stage 3: HTML/Code — 정밀 완성

**목적: Stage 2의 구조를 "프로덕션 퀄리티"로 끌어올린다. 여기서 디자인이 완성된다.**

Figma는 "대략 이 위치"까지 → 코드는 "정확히 이 위치". 한글 타이포그래피는 코드에서 다뤄야 제대로 된다.

#### Figma → Code 변환 프로세스

**"코드로 다듬어줘" 또는 Figma URL 공유 시 자동 실행:**

```
1. Figma MCP (get_design_context) → 디자인 읽기
2. 구조/컬러/서체/배치 자동 파악
3. HTML+CSS 코드로 변환
4. 패키지 비주얼 가이드와 대조 → 부족한 부분 자동 보강
5. 배경 이미지 삽입 (아래 이미지 규칙 참조)
6. 로컬 HTML 파일 생성 → 로컬 HTTP 서버 → 브라우저 프리뷰
```

**⚠️ 이미지 파일 규칙 (실전 검증됨):**
- 대화에 첨부된 이미지는 직접 파일로 저장 불가 → **대표님에게 Downloads 폴더 저장 요청**
- 저장 확인: `find ~/Downloads -name "파일명*"` 로 검색
- HTML에서 사용: 이미지를 `docs/` 폴더로 복사 → 상대 경로로 참조
- 로컬 프리뷰: `python3 -m http.server 8787` → `http://localhost:8787/` (file:// 프로토콜은 browse 도구에서 차단됨)
- 폰트 비교 요청 시: 동일 레이아웃 + 폰트만 다른 HTML 2개를 상하로 배치해서 비교 페이지 생성

#### Figma에서 읽는 것 vs 코드에서 보강하는 것

| 항목 | Figma에서 읽기 | 코드에서 보강 |
|------|--------------|-------------|
| 레이아웃 | 요소 위치, 크기, 간격 | CSS Grid/Flexbox 정리 |
| 타이포 | 서체, 크기, 굵기 | 한글 자간/행간 미세 조정, 웹폰트 |
| 컬러 | HEX 값 | CSS 변수화, 그라데이션 정밀 제어 |
| 이미지 | 배치 위치, 크기 | object-fit, 오버레이, 블러 효과 |
| 효과 | 그림자, 블러 | CSS filter, backdrop-filter 정밀값 |
| 텍스트 | 내용, 기본 스타일 | text-shadow, gradient text, 강조 효과 |

#### 코드에서만 가능한 디테일

| 기능 | 설명 | 예시 CSS |
|------|------|---------|
| 텍스트 그라데이션 | 제목에 2색 그라데이션 | `background: linear-gradient(); -webkit-background-clip: text;` |
| 배경 오버레이 | 이미지 위 반투명 컬러/그라데이션 | `background: linear-gradient(rgba(0,0,0,0.6), ...)` |
| 한글 타이포 정밀 | 자간, 행간, 단어 간격 | `letter-spacing: -0.02em; line-height: 1.35;` |
| 배경 블러 | 텍스트 뒤 유리 효과 | `backdrop-filter: blur(12px);` |
| 다중 그림자 | 깊이감 있는 텍스트 | `text-shadow: 0 2px 4px rgba(), 0 4px 12px rgba();` |
| 정확한 px 제어 | 1px 단위 간격, 정렬 | `padding: 56px 64px; gap: 16px;` |

#### 수정 사이클 (Stage 3의 핵심 가치)

대표님의 수정 요청에 **즉시** 대응한다. Figma를 다시 열 필요 없다.

| 대표님 | Claude |
|--------|--------|
| "제목 좀 더 크게" | `font-size: 52px → 58px` 변경 → 새로고침 |
| "숫자 색 좀 더 밝게" | `color: #E11D48 → #F43F5E` 변경 → 즉시 확인 |
| "간격 넓혀줘" | `gap: 16px → 24px` 변경 → 즉시 확인 |
| "배경 좀 더 어둡게" | 오버레이 `opacity: 0.6 → 0.75` → 즉시 확인 |
| "다른 Midjourney 이미지로 바꿔볼래" | `background-image` URL 교체 → 즉시 확인 |
| "OK 이걸로 뽑아줘" | Puppeteer → PNG 저장 |

#### 최종 Export

| 용도 | 포맷 | 방법 |
|------|------|------|
| 블로그 업로드 | PNG 1200x630 | HTML → Puppeteer 스크린샷 |
| 인스타그램 | PNG 1080x1080 | 동일 (사이즈 변경) |
| 유튜브 썸네일 | PNG 1280x720 | 동일 |
| 웹 삽입 | HTML | 코드 그대로 사용 |
| 고해상도 인쇄 | PNG @2x | 2400x1260으로 스크린샷 |

**"PNG로 뽑아줘" 트리거 시**: HTML 파일을 Puppeteer로 정확한 사이즈 스크린샷 → PNG 저장.

### 3-Stage 전체 흐름 요약

```
패키지 완성 → "만들어줘"
  ↓
Stage 1: Midjourney — 시각적 방향 탐색 (~5분)
  ├── 3트랙(포토/그래픽/컨셉) x 4장 = 12장 프롬프트 제공
  ├── 대표님: Midjourney에서 생성 → "이 느낌!" 선택
  └── 산출물: 배경 이미지 + 색감/구도/무드 확정
  ↓
Stage 2: Figma — 레이아웃 확정 (~5분)
  ├── Claude: MCP로 초기 레이아웃 자동 생성
  ├── 대표님: 이미지 삽입 + 텍스트 배치 드래그 조정
  └── 산출물: 확정된 구조 (Figma URL)
  ↓
Stage 3: HTML/Code — 정밀 완성 (~5분)
  ├── Claude: Figma 읽기 → HTML 변환 → 디테일 보강
  ├── 대표님: "크게/밝게/넓게" → Claude 즉시 수정
  └── 산출물: 최종 PNG 파일
```

**숏컷 (스테이지 스킵):**
- 이미 Figma에서 만든 게 있을 때: **Stage 3부터** — "이 피그마 코드로 다듬어줘"
- 이미지 없이 타이포 중심: **Stage 1 그래픽 트랙 → Stage 2 → 3**
- 급할 때: **Stage 1 스킵 → Stage 2 → 3** (CSS 배경/그라데이션으로 대체)
- 초급할 때: **Stage 3만** — "HTML로 바로 만들어줘" (CSS만으로 완성)

## Failure Recovery

상황이 막혔을 때의 탈출 전략.

| 상황 | 대응 |
|------|------|
| **요청이 너무 애매함** | 가장 임팩트 큰 질문 1개만 던진다 — "누가 보는 자료인가요?" |
| **빠른 결과 원함** | 가정을 명시하고 드래프트 먼저 제공 — "[X로 가정하고 만들었습니다]" |
| **목표가 서로 충돌** | 충돌을 짧게 설명 + 2가지 경로 제시 + 하나 추천 |
| **전략 없이 실행 요구** | 간단한 planning core(원라이너+구조)를 먼저 만들고 실행 연결 |

## Domain Awareness

도메인에 따라 설득 논리가 다르다. 다음 도메인이 감지되면 자동 적응.

| 도메인 | 핵심 설득 논리 | 증거 유형 |
|--------|--------------|----------|
| 부동산/임대 | 수익률, 안정성, 실전 경험 | 숫자(공실률, 수익률), before/after |
| 교육/코칭 | 변화 약속, 커리큘럼 체계 | 수강생 후기, 결과물 사례 |
| 뷰티/웰니스 | 감성적 변화, 비포/애프터 | 사진, 인증 후기, 성분/기술 |
| B2B SaaS | ROI, 시간 절약, 통합 | 도입 사례, 데모, 비교표 |
| 전문서비스 | 신뢰, 전문성, 실적 | 자격, 수상, 고객사 로고 |

해밀시아 부동산 도메인이 감지되면 자동으로 수익률/공실률/실전경험 논리를 우선 적용.

## Common Mistakes

| 실수 | 올바른 접근 |
|------|-----------|
| 질문을 10개 이상 한꺼번에 던짐 | 복잡도 게이트 기준, 한 번에 하나씩, 선택지 제공 |
| 카피 없이 구조만 제공 | 구조 + 카피 + 비주얼 가이드를 항상 세트로 (G2) |
| 모든 결과물에 같은 포맷 적용 | 문서/랜딩/이미지/SNS별 패키지 포맷 분리 |
| 디자인을 직접 만들려고 함 | 기획 패키지까지만 → 실행은 전문 스킬로 핸드오프 |
| 참고자료를 무시하고 처음부터 시작 | 참고자료 있으면 분석 → 질문 축소 → 빠른 확인 (G1) |
| "알아서 해줘"를 거부함 | 합리적 기본값 세팅 후 결과 제시 → 피드백 받기 |
| 수정 요청에 전체 재기획 | 진단 → 유지/수정/전환 분류 → 필요한 엔진만 실행 |
| 원라이너 빠뜨림 | 원라이너가 모든 카피의 방향타 — 생략 금지 (G5) |
| 비주얼 가이드 스킵 | 문서든 블로그든 비주얼 엔진 필수 실행 (G2) |

## Quick Reference

```
Phase 0: 라우팅
  입력 → 경로 분기 (신규/수정/핸드오프) → 복잡도 (Fast/Standard/Extended)

Phase 1~4: 신규 기획 (메인 경로)
  감지 → Essentials 체크 → 선별 질문 → 4엔진 → 패키지 → Quality Gate

Phase R: 수정 기획 (수정 경로)
  초안 접수 → 5가지 진단 → 유지/수정/전환 분류 → 엔진 부분 실행 → 수정 패키지

Phase H: 핸드오프 (핸드오프 경로)
  기획 결과 → 역할 감지 → 역할별 브리프 분리 출력

Phase E: 3-Stage 디자인 파이프라인 ("만들어줘" 트리거)
  Stage 1: Midjourney — 3트랙 12장으로 시각적 방향 탐�� (색감/구도/무드)
  Stage 2: Figma — MCP 초기 생성 + 대표님 드래그 조정 → 레이아웃 확정
  Stage 3: Code — Figma 읽기 → HTML 변환 → 정밀 완성 → PNG Export
  숏컷: Figma 있으면 Stage 3부터 / 급하면 Stage 3만 (HTML 직행)

공통:
  G1: 참고자료 있으면 질문 축소
  G2: 비주얼 엔진 필수 (수정 경로 제외)
  G5: 원라이너 생략 금지
  Quality Gate: 8항목 자동 점검
```
