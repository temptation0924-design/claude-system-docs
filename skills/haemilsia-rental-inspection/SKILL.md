## 2중 검증 체계 — 간편점검 vs 빡센점검 (v2.0)

> **같은 점검을 두 가지 레벨로 검증한다.**

| 이름 | 버전 | 실행 주체 | 실행 방식 | 체크 수준 | 언제 |
|------|------|---------|----------|----------|------|
| **간편점검** | v1.0 | Railway 봇 (`haemilsia-bot`) | 매일 **07:30 KST 자동** | 7개 DB 기본 규칙 판정 | 매일 아침 |
| **빡센점검** | v2.0 | Claude Code (대표님) | "검증해줘" / "임대점검해줘" 명령 | 7개 DB + 29항목 체크리스트 + 95% 스코어링 | 필요할 때 |

### 역할 분담

**간편점검 (v1.0, 봇 자동)** — 매일 돌아가는 "방범 순찰"
- 7개 DB 쿼리 → 기본 규칙 판정 → 슬랙 알림
- 봇이 돌아갔는지 + 알림이 갔는지 보장
- 실패 시 봇 로그로 확인
- **트리거**: 매일 07:30 자동 or 슬랙에서 `전체현황` 입력

**빡센점검 (v2.0, 수동 전체)** — 정기 "정밀 검사"
- 29항목 체크리스트 + 95% 스코어링
- DB 쿼리 폴백 전략 (3단계)
- 자동 복구 루프 (최대 2회)
- Stop 훅 + 슬랙 전송 차단 훅으로 강제
- 점검보고서 DB 등록 + 미해결 추적
- **트리거**: Claude Code에서 "임대점검해줘" / "검증해줘" / "빡센점검해줘"
- → 이 섹션 아래부터 정의됨

### 원칙
- **봇은 건드리지 않는다** — 봇이 하는 간편점검(v1.0)은 기존 방식 그대로 유지
- **빡센점검(v2.0)은 수동 트리거 전용** — Railway 자동 실행에 통합하지 않는다
- **간편점검과 빡센점검의 결과가 다를 수 있다** — 빡센점검이 더 꼼꼼하므로 더 많은 오류를 찾을 수 있음 (정상)

### 언제 어느 걸 쓰나?

| 상황 | 추천 |
|------|------|
| 매일 아침 확인 | 간편점검 (봇이 자동으로 슬랙에 보내줌) |
| 중요한 결정 전 | 빡센점검 ("빡센점검해줘" → 29항목 체크) |
| 데이터 정합성 의심 | 빡센점검 |
| 주간/월간 정기 감사 | 빡센점검 |
| 신규 임차인/퇴거 발생 직후 | 빡센점검 |

| 제약 | 설명 | 대응 |

## DB별 점검 규칙

### 1️⃣ 임차인마스터 — 12규칙 API 판정 (v2.0 추가)

전체 쿼리 (사건종료=false) → 12규칙을 원시 필드 + relation 유무로 직접 판정.
rollup 필드가 omitted되므로, relation 필드의 빈값/값있음으로 건수(>0 vs ==0)를 대체.
거주상태/퇴거정산상태 등 rollup 값이 필요한 경우 → 관련 레코드를 fetch하여 직접 읽기.

**판정에 필요한 필드**:
| 필드 | 소스 | API 접근 방식 |
|------|------|-------------|
| 이름👤, 건물👤, 호수👤 | 임차인마스터 직접 | 직접 읽기 |
| 계약종료 (checkbox) | 임차인마스터 직접 | 직접 읽기 |
| 📄 DB#계약서 (relation) | 임차인마스터 직접 | 빈값 체크 |
| 3️⃣이사예정관리 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 4️⃣공실검증 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 6️⃣아이리스공실 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 7️⃣퇴거정산서 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 2️⃣미납리스크 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 거주상태👤 | 이사예정관리 DB | relation ID로 fetch → 거주상태👤 읽기 |
| 퇴거상황 | 퇴거정산서 DB | relation ID로 fetch → 퇴거상황 읽기 |

**판정 순서** (Notion 수식과 동일한 우선순위):

```
1. 이름👤 빈값 → ❌ 이름 미입력
2. 건물👤 빈값 → ❌ 건물 미입력
3. 호수👤 빈값 → ❌ 호수 미입력
4. 계약종료=false AND 계약서 relation 빈값 → ❌ 계약서 미연결

--- 이사예정 있는 경우 ---
5. 이사예정 있음 AND 거주상태=추가연장X → ✅ 정상:추가연장X
6. 이사예정 있음 AND 거주상태=거주중 AND 아이리스공실 있음 → ✅ 정상:기간내퇴거
7. 이사예정 있음 AND 거주상태=거주중 AND 아이리스공실 없음 → ❌ 조건부퇴거_아이리스미등록

--- 이사예정+공실검증 동시 (규칙11 복합 분기) ---
8.  이사예정 있음 AND 공실검증 있음 AND 퇴거정산서 없음 → ❌ 공실전환_이사예정미해제+퇴거정산서미작성
9.  이사예정 있음 AND 공실검증 있음 AND 퇴거상황≠확인완료 → ⚠️ 공실전환_이사예정미해제(정산진행중)
10. 이사예정 있음 AND 공실검증 있음 AND 계약종료=false → ❌ 공실전환_이사예정미해제+계약종료미처리
11. 이사예정 있음 AND 공실검증 있음 → ❌ 공실전환_이사예정미해제

--- 이사예정만 있는 경우 ---
12. 이사예정 있음 AND 퇴거정산서 없음 → ❌ 퇴거정산서 미작성
13. 이사예정 있음 AND 퇴거상황≠확인완료 → ⚠️ 퇴거정산 진행중
14. 이사예정 있음 AND 계약종료=false → ❌ 계약종료 미처리

--- 신규입주 (이사예정 없음 + 공실검증 없음) ---
15. 이사예정 없음 AND 공실검증 없음 AND (미납 or 아이리스 or 퇴거정산 있음) → ❌ 신규입주_불필요링크존재

--- 공실검증만 있는 경우 ---
16. 공실검증 있음 AND 퇴거정산서 없음 → ❌ 퇴거정산서 미작성
17. 공실검증 있음 AND 퇴거상황≠확인완료 → ⚠️ 퇴거정산 진행중
18. 공실검증 있음 AND 계약종료=false → ❌ 계약종료 미처리

19. 위 모두 해당 없음 → ✅ 정상
```

**구현 순서**:
1. 임차인마스터 전체 쿼리 (사건종료=false, pagination 적용)
2. 각 레코드의 relation 필드 빈값/값있음 확인
3. 이사예정 relation 있는 레코드 → 해당 이사예정관리 페이지 fetch → 거주상태👤 읽기
4. 퇴거정산서 relation 있는 레코드 → 해당 퇴거정산서 페이지 fetch → 퇴거상황 읽기
5. 위 판정 순서대로 첫 번째 매칭되는 규칙으로 결과 확정 (Notion 수식과 동일한 ifs 우선순위)
6. 오류/경고 건 → 점검보고서 등록 (중복방지 규칙 적용)

**예외 처리**:
- 템플릿 레코드 (`_000_윤실장`) → 판정 제외
- relation fetch 시 404 → Ghost Link로 판정, "유령링크" 오류 등록
- API rate limit → 0.35초 간격 호출 (공통 핸들링 적용)

### 2️⃣ 미납리스크
🚨[연체]확인필요 뷰 사용 가능 (checkbox + formula 복합 필터이나 작동 확인됨).
원시 필드로 충분:
- `대응단계👤` → 긴급대처/상습연체자/입금대기중 등 분류
- `누적미납액👤` → 금액 표시
- `연체입금예정일👤` → 오늘 날짜와 비교 → ⚡오늘 입금예정 표시
- `연체횟수👤` → 연체 횟수 표시

### 3️⃣ 이사예정관리
**오류 뷰 사용 금지** — formula 필터가 API에서 작동 안 함 (0건 반환).
📊전체이력 뷰 (완료=false) 쿼리 → Python에서 직접 체크:
- `만기일👤` < 오늘 + `거주상태👤` = 거주중 → ❌ 자동연장 처리 필요
- `거주상태👤` = 추가연장X + `이사예정일👤` ≤ 오늘 → ❌ 퇴실점검 필요

### 4️⃣ 공실검증
📊전체이력 뷰 (확인=false) 쿼리 → Python에서 직접 체크:
- `공실사유👤` 미입력 → ❌ 오류
  - **예외: `임대가능여부👤` = 특별업무수행건은 제외** (단기임대 입주 중이라 공실사유 불필요)
- `아이리스공실` relation 미연결 → ❌ 오류

### 3️⃣ 이사예정관리 — 추가연장X 아이리스공실 연결 규칙
- `거주상태=추가연장X` AND `만기일 - 오늘 ≤ 30일` → 아이리스공실 **연결 필수** (미연결 = 오류)
- `거주상태=추가연장X` AND `만기일 - 오늘 > 30일` → 미연결 = **정상**
- 거주중/추가연장X에서 아이리스공실 미연결 자체는 이사예정관리 점검 규칙이 아님 (위 30일 규칙만 적용)

### 6️⃣ 아이리스공실
**핵심: 크로스 DB 비교 필수**
📊전체확인 뷰 (확인=false) 쿼리 + 공실검증 DB도 함께 쿼리.
Python에서 직접 체크:
- `아이리스상태👤` vs 공실검증의 `임대가능여부👤` 비교 → 불일치 시 ❌ 상태불일치
- 퇴거날짜지정→기간내퇴거 → 상태 업데이트 필요 (이대리 담당)

**⚠️ 기간내퇴거 예외규칙 (2026-04-09 추가):**
- 아이리스=기간내퇴거 AND 공실검증=기간내퇴거 → **정상** (점검보고서 미등록)
- 아이리스=기간내퇴거 AND 공실검증=일반임대 가능 → **불일치 오류** (점검보고서 등록)
- 즉, 공실검증에도 기간내퇴거로 되어 있으면 아직 퇴거 진행 중이므로 "일반임대 전환 필요" 오류가 아님

**오류가 아닌 것:**
- 임차인마스터 미연결 → 공실이면 임차인 없는 게 정상
- 퇴거날짜지정/기간내퇴거 상태에서 공실검증 미연결 → 아직 퇴거 전이라 정상
- 아이리스=기간내퇴거 AND 공실검증=기간내퇴거 → 퇴거 진행 중이므로 정상

#### 📋아이리스[엑셀] vs 아이리스상태👤 자동 비교 (v2.0 Phase 4 추가)

> **🔄최신화검증 formula가 API에서 읽히지 않으므로, 직접 비교한다.**

**비교 전 문자열 정규화** (C2 해결):
- 양쪽 값 비교 전 반드시 정규화 적용: `strip()` + 연속공백→단일공백 + 이모지 제거
- 예: "📑예약중 " → "예약중", " 일반임대  가능" → "일반임대 가능"
- Notion select 필드는 정의된 옵션만 반환하므로 실제 변형은 드물지만, 방어적으로 적용

**정방향 비교** (DB 레코드 기준 — 자동 실행):
아이리스공실 DB 쿼리 시 `📋아이리스[엑셀]` + `아이리스상태👤` 두 필드를 함께 가져와 비교.

| 📋아이리스[엑셀] | 아이리스상태👤 | 판정 | 조치 |
|-----------------|--------------|------|------|
| 동일 | 동일 | ✅ 정상 | 없음 |
| 불일치 | 불일치 | ❌ 상태불일치 | 점검보고서 등록 (오류유형: 상태오류, 긴급도: 긴급, 담당: 이대리) |
| 빈값 | 값있음 | ⚠️ 엑셀 미입력 | 점검보고서 등록 (오류유형: 미입력, 긴급도: 참고, 담당: 이대리) |
| 값있음 | 빈값 | ⚠️ 상태 미입력 | 점검보고서 등록 (오류유형: 미입력, 긴급도: 확인, 담당: 이대리) |
| 둘 다 빈값 | 둘 다 빈값 | ⚠️ 양쪽 미입력 | 점검보고서 등록 (오류유형: 미입력, 긴급도: 참고, 담당: 이대리) |

※ 오류유형 구분 (W1 해결):
- 엑셀 vs 상태 불일치 → 오류유형 = `상태오류` (기존 공실검증 "불일치"와 구분)
- 기존 아이리스상태 vs 공실검증 임대가능여부 불일치 → 오류유형 = `불일치` (기존 유지)

**불일치 예외 처리**:
- 기간내퇴거 예외: 아이리스[엑셀]=기간내퇴거 AND 아이리스상태👤=기간내퇴거 → 정상 (기존 예외규칙과 동일)
- 퇴거날짜지정→기간내퇴거 전환 중: 한쪽만 변경된 경우 → 상태오류로 등록하되 긴급도="확인" (전환 과정)

**역방향 비교** (수동 실행 전용 — C1 해결):
> **Railway 자동 실행(07:05)에서는 실행하지 않는다. 대표님이 엑셀 이미지를 제공하여 수동으로 점검을 트리거할 때만 실행.**

엑셀 이미지에서 추출한 호실 목록 vs 아이리스공실 DB 전체 레코드를 건물+호수로 대조.

| 상황 | 판정 | 조치 |
|------|------|------|
| 엑셀에만 있는 호실 | ⚠️ DB 미등록 | 점검보고서 등록 (오류유형: 미연결, 긴급도: 참고, 담당: 윤실장) |
| DB에만 있는 호실 | ⚠️ 엑셀 미반영 | 점검보고서 등록 (오류유형: 미연결, 긴급도: 참고, 담당: 이대리) |
| 양쪽 모두 있음 | ✅ 정상 | 없음 |

**역방향 비교 실행 조건**:
- 수동 점검 시 엑셀 이미지가 제공된 경우에만 실행
- 자동 점검(Railway) 시 → 역방향 비교 자동 스킵 (슬랙에 별도 표시 안 함)
- 수동 점검 시 엑셀 이미지 미제공 → "📋 엑셀 이미지 미제공 — 역방향 비교 생략" 표시
- 엑셀 이미지에서 호실 추출 실패 시 → 역방향 비교 스킵 + 점검보고서에 참고 등록

## 미해결 항목 추적 (v2.0 Phase 5 추가)

> **점검보고서 DB의 미완료 항목을 경과일 기준으로 추적하고, 슬랙에 리마인더를 보낸다.**

### 조회 대상
- 점검보고서 DB에서 `☑️처리완료=false` AND `처리상태≠"전송실패"` 항목 전체 조회
- 각 항목의 `점검일` 필드로 경과일 계산: `오늘 - 점검일 = 경과일`

**경과일 계산 방어 처리** (C3 해결):
- 점검일 빈값 → 해당 항목 경과일 계산 스킵 + 슬랙에 `⚠️ [점검일 미입력] Name — 점검일 확인 필요` 표시
- 점검일이 미래 날짜 (경과일 < 0) → 해당 항목 스킵 + 점검보고서에 `참고` 등급으로 "점검일 미래 날짜 오류" 등록
- 경과일 계산 실패 시 → 항목별 스킵 (전체 스크립트 중단 금지)

### 에스컬레이션 경로

| 경과일 | 등급 | 슬랙 조치 | 점검보고서 DB 조치 |
|--------|------|----------|------------------|
| 1일 | 알림 | 미해결 섹션에 `⏰` 표시 | 없음 |
| 3일 | 리마인더 | 담당자 이름 **굵게** + "3일 경과" 강조 | 없음 |
| 7일 | 에스컬레이션 | `🔴` 표시 + 대표님 멘션 + 긴급도 자동 승격 | 긴급도 1단계 승격 (1회만) |
| 14일 | 최종 경고 | 대표님 DM 별도 전송 (통합 1건) | `장기미해결` 태그 추가 + 긴급도 최고 승격 |

### 긴급도 자동 승격 규칙

**멱등성 보장** (C1 해결):
- 승격은 **경과일 임계값을 처음 넘는 시점에 1회만** 실행
- 판정 기준: 현재 긴급도가 승격 대상인지 확인 → 이미 승격 완료 상태면 스킵
  - 7일 경과 + 긴급도=참고 → 확인으로 승격 (실행)
  - 7일 경과 + 긴급도=확인 → 이미 승격 완료 또는 원래 확인 → 스킵 (긴급 승격은 14일에)
  - 7일 경과 + 긴급도=긴급 → 스킵
  - 14일 경과 + 긴급도≠긴급 → 긴급으로 승격 (실행)
  - 14일 경과 + 긴급도=긴급 → 스킵 (DM만 발송)
- DB UPDATE 전 현재 값 확인 → 변경 불필요 시 API 호출 생략 (rate limit 절약)

### 담당자별 알림 방식

| 담당자 | 1~6일 | 7일+ | 14일+ |
|--------|-------|------|-------|
| 윤실장 | 슬랙 메시지 내 이름 표기 | 이름 **굵게** + 🔴 | 동일 |
| 이대리 | 슬랙 메시지 내 이름 표기 | 이름 **굵게** + 🔴 | 동일 |
| 대표님 | 슬랙 메시지 내 이름 표기 | 슬랙 채널 멘션 | **통합 DM 1건** 전송 |

**대표님 DM 설정** (C4 해결):
- 대표님 Slack User ID → `env-info.md`에 정의된 값 참조
- 14일+ 항목이 N건이어도 **통합 DM 1건**으로 전송 (알림 폭격 방지)
- DM 형식: `🔴 장기미해결 N건: [항목1], [항목2]... 점검보고서 확인 필요`

### 장기미해결 태그 (C5 해결)
- **저장 필드**: 점검보고서 DB의 `이유` text 필드 끝에 `[장기미해결]` 접미사 추가
- 별도 필드 신설 불필요 — 기존 text 필드 활용
- 이미 `[장기미해결]` 접미사가 있으면 중복 추가하지 않음 (멱등)

### 슬랙 미해결 섹션 형식
- 경과일 **내림차순** 정렬 (긴급한 것 먼저)
- **최대 20건**까지 표시, 초과 시 `... 외 N건 (점검보고서 참조)` + 보고서 링크 (C2 해결)
```
━━━ 📌 미해결 항목 (N건) ━━━
🔴 [14일] S동_101 미납 — 담당: 대표님 ← 최종경고·DM발송
🔴 [7일] A동_404 계약서 미연결 — 담당: **윤실장** ← 에스컬레이션
⏰ [3일] FLAT2_203 공실검증 미연결 — 담당: **이대리** (리마인더)
⏰ [1일] FLAT2_301 상태미입력 — 담당: 윤실장
... 외 5건 (<https://www.notion.so/a74d6ce07341401c88ff57e68063d6bf|점검보고서 전체 보기>)
```

### 미해결 0건 시
- 미해결 항목이 없으면 미해결 섹션 자체를 표시하지 않음 (깔끔한 알림 유지)
- 요약 라인에만 `📌미해결 0건` 표시

### 처리완료 시 에스컬레이션 해제
- `☑️처리완료=true` 체크 시 → 다음 점검부터 미해결 섹션에서 자동 제외
- 승격된 긴급도는 원복하지 않음 (이력 보존)

### Phase 3 동시 실행과의 관계
- 미해결 항목 조회는 [2단계] 전송실패 복구 이후, [3단계] 중복 필터링과 동시에 실행 가능
- 미해결 항목 정보는 [4단계] 슬랙 메시지 조합 시 포함
- 긴급도 자동 승격(DB 업데이트)은 [4단계] B(점검보고서 DB 등록)와 함께 실행
- 승격 UPDATE 시 공통 rate limit 핸들링 적용 (0.35초 간격)

> **슬랙 알림 후 반드시 점검보고서 DB에 오류 항목을 기록한다.**

| 항목 | 값 |
|------|-----|
| DB 이름 | `0️⃣ 🤖 점검보고서_Haemilsia[임대]` |
| Notion ID | `a74d6ce07341401c88ff57e68063d6bf` |
| Data Source ID | `c63b4a3d-360f-4ec4-8e41-bf34a405e248` |

### 기록 대상
- 6개 DB 점검에서 발견된 **오류 항목만** 기록 (정상 항목은 기록하지 않음)
- 미납리스크는 별도 채널에서 관리하므로 점검보고서에는 **오류 항목만** 등록

### 중복 방지 규칙 (v2.0 추가)

> **같은 오류를 매일 중복 등록하지 않는다.**

**실행 순서**:
1. 점검보고서 DB에서 `☑️처리완료=false` 항목 전체 조회
2. 각 미완료 항목의 비교키 생성
3. 새 오류 발견 시 → 비교키 매칭 → 이미 등록되어 있으면 스킵
4. 새 오류만 점검보고서에 등록

**비교키 생성 규칙**:
- 비교키 = `{건물}_{호수}_{오류유형}` (3필드 복합키)
- 건물명 정규화: 공백 제거, 대소문자 통일 (예: "FLAT 2" → "FLAT2", "S동" → "S동")
- 호수 정규화: 원본 유지, 단 "호" 접미사 제거 (예: "203호" → "203", "B1" → "B1")
- 오류유형: 7개 중 하나 (미입력/불일치/날짜경과/미납/미연결/상태오류/유령링크)

**매칭 예시**:
| 기존 등록 | 새 발견 | 비교키 | 결과 |
|-----------|---------|--------|------|
| FLAT2_203_미연결 | FLAT 2_203호_미연결 | FLAT2_203_미연결 | 스킵 (중복) |
| FLAT2_203_미연결 (처리완료=true) | FLAT2_203_미연결 | - | 등록 (재발) |
| FLAT2_203_미연결 | FLAT2_203_미입력 | 다름 | 등록 (별도 오류) |

## 실행 워크플로우 (v2.0 Phase 3 추가)

> **점검→판정→{슬랙+보고서} 동시 실행. 순차 실행하지 않는다.**

### 실행 순서

```
[1단계] 점검 실행
  6개 DB 쿼리 → 규칙 판정 → 오류 목록 생성

[2단계] 전송실패 복구 (중복필터보다 선행) ← C1 수정
  점검보고서 DB에서 처리상태="전송실패" 레코드 조회
  → 미전송 건 재시도 (실패채널 필드 기준으로 슬랙/DB 중 실패한 쪽만)
  → 성공 시 처리상태="시작 전" + Name 접두사 제거
  ※ 복구 완료된 레코드는 [3단계]에서 정상 미완료 항목으로 인식됨

[3단계] 중복 필터링
  점검보고서 DB 미완료 항목 조회 (처리상태≠"전송실패") → 비교키 매칭
  ※ 비교키는 Name 파싱이 아닌 건물+오류유형 별도 필드 기반으로 생성
  → 신규 오류만 추출

[4단계] 동시 실행 ← 핵심 변경
  ┌─ A. 슬랙 메시지 조합 + 전송 (점검보고서 링크 포함)
  │
  ├─ B. 점검보고서 DB 등록 (소스페이지URL 포함)
  │
  └─ 두 작업을 동시에 실행. 순서 의존성 없음.
  ※ 오류 0건인 경우에도 "이상 없음" 슬랙 메시지 전송 (미실행과 구분)
```

**중복방지 ↔ 전송실패 충돌 방지 규칙** (C1 해결):
- [2단계] 전송실패 복구가 [3단계] 중복 필터보다 **반드시 먼저** 실행
- 복구 성공 → `처리상태="시작 전"`으로 변경 → 3단계에서 정상 미완료 항목으로 인식
- 중복 필터는 `처리상태="전송실패"` 레코드를 비교 대상에서 **제외** (아직 미복구된 건)
- 비교키는 `건물` select 필드 + `오류유형` select 필드 기반 (Name 파싱 아님)

### 슬랙 메시지에 점검보고서 링크 포함

슬랙 알림 하단에 점검보고서 DB 바로가기 링크를 추가:
```
━━━━━━━━━━━━━━━━━━━
오류 N건 | 미납 N건 | ⚡오늘 입금예정 N건
📋 점검보고서: https://www.notion.so/a74d6ce07341401c88ff57e68063d6bf
```

개별 오류 항목에도 소스 DB 레코드 URL 포함 (Slack mrkdwn 형식):
```
❌ [FLAT2_203_이름] — 공실검증 미연결 📎<https://notion.so/소스페이지URL|소스 페이지>
```
※ `📎(URL)` 형식은 Slack에서 링크로 인식 안 됨 → 반드시 `<URL|텍스트>` mrkdwn 형식 사용
※ 여기서 URL은 소스 DB(임차인마스터 등) 레코드의 Notion URL이며, 점검보고서 레코드 URL이 아님

### 부분 실패 재시도 전략

| 상황 | 조치 |
|------|------|
| 슬랙 실패 + DB 성공 | 슬랙만 재시도 (최대 3회, 간격 5초) |

**다음 점검 시 전송실패 복구 흐름**:
1. 점검 시작 시 `처리상태="전송실패"` 레코드 조회
2. `실패채널` 필드 확인 → 실패한 쪽만 재시도
3. 성공 시 → `처리상태="시작 전"` + `Name` 접두사 제거 + `실패채널` 비우기
4. 재실패 시 → 유지 (다음 점검에서 다시 시도)

**Notion DB 자체 장애 시 폴백 (C2 해결)**:
```
[1차] Notion DB에 전송실패 레코드 등록 시도
  ↓ 실패
[2차] 슬랙 #haemilsia-task에 에러 메시지 전송 시도
  "⚠️ [점검시스템] Notion DB 장애로 전송실패 기록 불가. 오류 N건 미저장."
  ↓ 실패
[3차] Railway 애플리케이션 로그에 ERROR 레벨로 출력 (stdout)
  → Railway 대시보드에서 로그 확인 가능 (유실되지 않음)
```

- 3단계 폴백 체인으로 최소 1곳에는 실패 기록 남김

**금지사항**:
- Railway `/tmp` 디렉토리에 실패 로그 **파일** 저장 금지 (재시작 시 유실)
- 실패 정보는 Notion DB → 슬랙 → Railway 로그 순서로 폴백

## DB 쿼리 폴백 전략 (v2.0 추가)

> **뷰 URL 실패 시 포기하지 않는다. 3단계 폴백으로 반드시 데이터를 가져온다.**

### 폴백 순서

| 단계 | 방법 | 실패 조건 |
|------|------|----------|
| 1차 | `notion-query-database-view` (뷰 URL) | "view not found" 또는 400 에러 |
| 2차 | `notion-search` + `data_source_url` (컬렉션 URL) | 검색 결과 0건 또는 에러 |
| 3차 | `notion-fetch` (DB ID) → 뷰 URL 재구성 → 1차 재시도 | fetch 자체 실패 |

### 대용량 결과 처리
- 결과가 파일로 저장된 경우 → **서브에이전트 즉시 분석** (백그라운드 방치 금지)
- 서브에이전트 결과 확인 전 다음 DB 점검으로 넘어가지 않음
- 서브에이전트 prompt에 반드시 "Read the file in sequential chunks until 100% read" 포함

### A카테고리 게이트
- 6개 DB 중 1개라도 쿼리 실패(3단계 폴백 전부 실패) → 규칙 판정 단계로 넘어가지 않음
- 실패한 DB 이름 + 에러 메시지를 대표님께 보고 → 수동 개입 요청

| C1 | 전송실패 복구 (2단계) 실행 | 처리상태=전송실패 조회 실행됨 (0건도 PASS) |
| C2 | 중복 필터링 실행 | 비교키 매칭으로 신규/기존 분류됨 |
| C3 | 신규 오류 점검보고서 등록 | 신규 오류 건수 = 등록 건수 (0건도 PASS) |
| C4 | 소스페이지URL 포함 | 등록된 모든 건에 URL 있음 |
| C5 | 중복 스킵 로그 표시 | 스킵 건 있으면 로그 출력됨 (0건이면 자동 PASS) |

### D. 미해결 추적 (5항목 x 3점 = 15점)

| # | 체크 항목 | PASS 조건 |
|---|---------|----------|
| D1 | 미완료 항목 조회 | ☑️처리완료=false AND 처리상태≠전송실패 쿼리 실행됨 |
| D2 | 경과일 계산 | 각 항목에 경과일(오늘-점검일) 산출됨 |
| D3 | 에스컬레이션 판정 | 7일/14일 임계값 체크 실행됨 |
| D4 | 긴급도 자동 승격 | 해당 건 있으면 승격 실행, 없으면 "해당없음"도 PASS |
| D5 | 경과일 내림차순 정렬 | 미해결 목록이 경과일 큰 순서로 정렬됨 |

### E. 슬랙 전송 (5항목 x 3점 = 15점)

| # | 체크 항목 | PASS 조건 |
|---|---------|----------|
| E1 | 6개 DB 섹션 모두 포함 | 임차인/미납/이사/공실/아이리스/퇴거 전부 있음 |
| E2 | 미해결 항목 섹션 포함 | 미해결 N건 섹션 있음 (0건이면 생략도 PASS) |
| E3 | 요약 라인 형식 준수 | "오류 N건 \| 미납 N건 \| ⚡오늘 입금예정 N건 \| 📌미해결 N건" |
| E4 | 점검보고서 링크 포함 | 하단에 DB 링크 URL 있음 |
| E5 | 슬랙 실제 전송 완료 | slack_send_message 호출 성공 |

### 스코어 계산

총점 = (A PASS수 x 3) + (B PASS수 x 4) + (C PASS수 x 4) + (D PASS수 x 3) + (E PASS수 x 3)
최대 = 18 + 32 + 20 + 15 + 15 = 100점

| 점수 | 판정 | 조치 |
|------|------|------|
| 95~100 | ✅ PASS | 점검 완료 인정 |
| 80~94 | ❌ RETRY | FAIL 항목 자동 복구 → 재검증 (최대 2회) |
| 0~79 | 🚨 MANUAL | 수동 개입 요청 |

**게이트 규칙**:
1. A카테고리 1개라도 FAIL → B~E 채점 불가, 폴백 재시도 먼저
2. E5 예외: 슬랙 전송은 대표님 승인 후 → E5 제외 97점 이상이면 전송 승인 요청

### 자동 복구 전략

| FAIL 항목 | 복구 방법 |
|-----------|----------|
| A1~A6 (DB쿼리) | 뷰URL → search API → DB스키마 재확인 순서로 폴백 |
| B1 (12규칙) | 쿼리 데이터 기반 규칙 순회 재실행 |
| B7 (크로스DB) | 아이리스+공실검증 데이터 매칭 실행 |
| B8 (엑셀비교) | 아이리스 데이터에서 두 필드 비교 실행 |
| C1~C5 (보고서) | 중복비교→등록→URL 파이프라인 일괄 실행 |
| D1~D5 (미해결) | 점검보고서 재조회→경과일→에스컬레이션 일괄 |
| E1~E4 (슬랙형식) | 메시지 재조합 |

### 재검증 루프

                    ↓ 미만
           FAIL 항목 자동 복구
                    ↓
2차 검증 → 재채점 → 95점 이상? → 완료
                    ↓ 미만
           2차 복구
                    ↓
3차 검증 → 재채점 → 95점 이상? → 완료
                    ↓ 미만
           수동 개입 요청 (FAIL 목록 제시)
