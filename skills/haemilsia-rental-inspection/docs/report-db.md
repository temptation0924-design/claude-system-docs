# 점검보고서 DB 작성 (필수)

> **슬랙 알림 후 반드시 점검보고서 DB에 오류 항목을 기록한다.**

| 항목 | 값 |
|------|-----|
| DB 이름 | `0️⃣ 🤖 점검보고서_Haemilsia[임대]` |
| Notion ID | `a74d6ce07341401c88ff57e68063d6bf` |
| Data Source ID | `c63b4a3d-360f-4ec4-8e41-bf34a405e248` |

## 기록 대상
- 7개 DB 점검에서 발견된 **오류 항목만** 기록 (정상 항목은 기록하지 않음)
- 미납리스크는 별도 채널에서 관리하므로 점검보고서에는 **오류 항목만** 등록

## 중복 방지 규칙 (v2.0 추가)

> **같은 오류를 매일 중복 등록하지 않는다.**

**실행 순서**:
1. 점검보고서 DB에서 `☑️처리완료=false` 항목 전체 조회
2. 각 미완료 항목의 비교키 생성
3. 새 오류 발견 시 → 비교키 매칭 → 이미 등록되어 있으면 스킵
4. 새 오류만 점검보고서에 등록

**비교키 생성 규칙**:
- 비교키 = `{건물}_{호수}_{오류유형}` (3필드 복합키)
- 건물명 정규화: 공백 제거, 대소문자 통일 (예: "FLAT 2" → "FLAT2", "S동" → "S동")
- 호수 정규화: 원본 유지, 단 "호" 접미사 제거 (예: "203호" → "203", "B1" → "B1")
- 오류유형: 7개 중 하나 (미입력/불일치/날짜경과/미납/미연결/상태오류/유령링크)

**매칭 예시**:
| 기존 등록 | 새 발견 | 비교키 | 결과 |
|-----------|---------|--------|------|
| FLAT2_203_미연결 | FLAT 2_203호_미연결 | FLAT2_203_미연결 | 스킵 (중복) |
| FLAT2_203_미연결 (처리완료=true) | FLAT2_203_미연결 | - | 등록 (재발) |
| FLAT2_203_미연결 | FLAT2_203_미입력 | 다름 | 등록 (별도 오류) |

**복합 오류 처리**: 한 레코드에서 오류 2건 이상 발견 시 → 각각 별도 비교키로 개별 등록

**특수 케이스 처리**:
- 복수 호실 통합 건 (예: "정다운건설_FLAT2+FLAT3_복수미납") → 비교키 = `{건물}_{Name 원본}_{오류유형}` (호수 추출 불가 시 Name 전체 사용)
- 비교키 생성 원본: `건물` select 필드 + `Name` title 필드에서 호수 추출 + `오류유형` select 필드

**슬랙 로그 형식**: 중복 스킵 시 슬랙 알림에 포함
```
📋 중복 스킵: FLAT2_203 미연결 (기존 미완료 항목 유지)
```

## 필드 매핑

| 필드 | 입력값 |
|------|--------|
| Name | `[건물]_[호수]_[오류요약]` |
| 점검일 | 점검 실행일 (오늘 날짜) |
| 소스DB | 오류가 발견된 DB명 (임차인마스터/미납리스크/이사예정관리/공실검증/아이리스공실/퇴거정산서/신규입주자DB) |
| 건물 | 해당 건물명 |
| 오류유형 | 미입력/불일치/날짜경과/미납/미연결/상태오류/유령링크 중 선택 |
| 긴급도 | 긴급/확인/참고 중 선택 |
| 담당자 | 윤실장/이대리/대표님 중 선택 |
| 이유 | 오류 상세 설명 |
| 할일 | 조치 방법 |
| 완료기준 | 어떤 상태가 되면 완료인지 |
| 처리상태 | `시작 전` (기본값) |

## 긴급도 판단 기준
- **긴급**: 상태전환 누락(날짜경과), 데이터 불일치, 긴급대처 미납
- **확인**: 미연결, 미입력, 일반 미납
- **참고**: 경미한 데이터 품질 이슈

## 담당자 판단 기준
- **윤실장**: 데이터 입력/수정 필요 (미입력, 미연결)
- **이대리**: 아이리스 상태 전환, 공실검증 업데이트
- **대표님**: 긴급대처 미납, 전략적 판단 필요

## 소스페이지URL 입력
- 점검보고서 등록 시 `소스페이지URL` 필드에 해당 오류 레코드의 Notion URL을 입력
- 클릭하면 바로 소스 DB의 해당 페이지로 이동 가능

## 윤실장 2단계 처리 프로세스 (2026-04-09 추가)

| 단계 | 처리상태 | ☑️처리완료 | 의미 |
|------|---------|-----------|------|
| 1단계 | 시작 전 | ❌ | 윤실장 아직 안 봄 |
| 2단계 | 진행 중 | ❌ | 윤실장 확인/처리 중 |
| 3단계 | 완료 | ❌ | 윤실장 처리 완료 → **대표님 확인 대기** |
| 4단계 | 완료 | ✅ | **대표님 최종 확인 → 종료** |

- 윤실장 뷰: `☑️처리완료 = false` 항목 표시 (윤실장이 볼 일감)
- 윤실장이 처리 완료 → `처리상태 = 완료`로 변경
- 대표님이 확인 후 → `☑️처리완료 = true` 체크 → 윤실장 뷰에서 자동 제외
- **핵심: 윤실장과 대표님 업무가 겹치지 않도록 2단계 확인 구조**
