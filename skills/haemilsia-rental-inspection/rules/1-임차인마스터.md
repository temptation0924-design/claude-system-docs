# 임차인마스터 — 26분기 수식 기반 판정 (v3.2)

전체 쿼리 (사건종료=false) → 25분기를 원시 필드 + relation 유무로 직접 판정.
rollup 필드가 omitted되므로, relation 필드의 빈값/값있음으로 건수(>0 vs ==0)를 대체.
거주상태/퇴거정산상태 등 rollup 값이 필요한 경우 → 파일 기반 URL 매칭으로 직접 읽기 (API 0번).

**판정에 필요한 필드**:
| 필드 | 소스 | API 접근 방식 |
|------|------|-------------|
| 이름👤, 건물👤, 호수👤 | 임차인마스터 직접 | 직접 읽기 |
| 사건종료 (checkbox) | 임차인마스터 직접 | 직접 읽기 |
| 📄 DB#계약서 (relation) | 임차인마스터 직접 | 빈값 체크 |
| 3️⃣이사예정관리 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 4️⃣공실검증 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 6️⃣아이리스공실 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 7️⃣퇴거정산서 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 2️⃣미납리스크 (relation) | 임차인마스터 직접 | 빈값=0건, 값있음=1건+ |
| 거주상태👤 | 이사예정관리 JSON | relation URL → 파일 매칭 → 거주상태👤 읽기 |
| 퇴거상황 | 퇴거정산서 JSON | relation URL → 파일 매칭 → 퇴거상황 읽기 |

**퇴거상황 종료 판정 기준**:
- `정산완료` = 정산 절차 완료 → 사건종료 대상
- `미연락대기` = 임차인이 받을 돈 없이 퇴거 완료, 연락 불가 → **임대종료건, 사건종료 대상** (정산완료와 동일 취급)

> 아래 규칙에서 `퇴거상황=종료상태`는 `정산완료` 또는 `미연락대기`를 의미함.

**구현 순서** (v3.0 파일 기반):
1. 임차인마스터 JSON 파일에서 전체 레코드 로드
2. 각 레코드의 relation 필드 빈값/있음 확인 (건수 대체)
3. 이사예정 relation URL → 이사예정관리 JSON에서 매칭 → 거주상태👤 읽기 (API 0번)
4. 퇴거정산서 relation URL → 퇴거정산서 JSON에서 매칭 → 퇴거상황 읽기 (API 0번)
5. 위 25분기 순서대로 첫 번째 매칭되는 규칙으로 결과 확정
6. 매칭 실패(relation URL이 있는데 JSON에 없음) → Ghost Link → "유령링크" 오류 등록

**예외 처리**:
- 템플릿 레코드 (`_000_윤실장`) → 판정 제외
- relation fetch 시 404 → Ghost Link로 판정, "유령링크" 오류 등록
- API rate limit → 0.35초 간격 호출 (공통 핸들링 적용)
- **🛡️ 특별업무수행건 예외 (v3.5, 2026-04-14)**: Name👤에 "특별업무수행" 포함된 임차인은
  공실검증 relation이 확인=true여도, Wave 1 쿼리 결과에 없어도 **유령링크로 판정하지 않는다**.
  이유: 특별업무수행건은 단기임대/위탁운영 형태로 공실검증 페이지가 확인=true 상태로 연결되어 계속 유지됨.
  이사예정 relation 완료=true에도 동일 예외 적용.
  → 판정 순서 rule 6/7(특별업무수행 분기)로 분기되어 ✅정상:특별업무수행로 종결됨.
