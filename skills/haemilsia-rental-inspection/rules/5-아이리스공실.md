# 아이리스공실 — 17분기 수식 기반 판정 (v3.4)

📊전체확인 뷰 (확인=false) 쿼리 → Python에서 직접 판정.

> **2026-04-14 변경 (v3.3)**: 규칙 3 "유령 relation 감지"에 **페이지삭제(진짜 dangling) 감지 분기** 추가. 임차인마스터 relation URL이 Wave 1 쿼리 결과(사건종료=false) 목록에 없을 때 개별 `notion-fetch`로 실존 확인 → 페이지 있으면 🗑️ 유령연결, 404면 🚨 유령링크:페이지삭제. 15→16분기.
>
> **2026-04-14 변경 (v3.4 → v3.7 수정)**: 새 rule 7 추가 — 아이리스가 **퇴거날짜지정** 상태인데 이사예정관리 DB에 등록되지 않은 케이스 감지. 이사예정 → 아이리스 forward는 임차인마스터 단계에서 잡지만, 아이리스 → 이사예정 reverse는 누락이었음. 16→17분기.
> **v3.7 (2026-04-14)**: 기간내퇴거는 "임차인 중도해지 → 이사예정 건너뛰고 바로 공실검증/아이리스" 플로우라 이사예정 빈 게 정상. rule 7에서 **기간내퇴거 제외**, **퇴거날짜지정만 오류 판정**. 대표님 직접 확정.

**판정에 필요한 필드**:
| 필드 | 타입 | 용도 |
|------|------|------|
| Name | title | 식별 |
| 아이리스상태👤 | select | 현재 상태 |
| 📋아이리스[엑셀] | select | 엑셀 기준 상태 (고정 필드명, 날짜 접미사 없음) |
| 입력일자👤 | date | 입력 날짜 |

**판정 순서**:

```
1. 아이리스상태 빈값 → 🚨 아이리스상태 미입력
2. 임차인마스터 relation ≥ 2개 → 🚨 임차인중복연결 (아이리스 페이지에 임차인 2명 이상 연결됨)
3. 유령 relation 감지 (하나라도 해당 시):
   - 공실검증 연결 있음 AND 연결된 공실검증 확인=true → 🗑️ 유령연결:공실검증(정리필요)
   - 이사예정 연결 있음 AND 연결된 이사예정 완료=true → 🗑️ 유령연결:이사예정(정리필요)
   - 임차인 연결 있음 AND 연결된 임차인마스터 사건종료=true → 🗑️ 유령연결:임차인(정리필요)
   - 임차인 연결 있음 AND Wave 1 쿼리 결과에 없음 → **개별 fetch 폴백**:
     - 페이지 존재 + 사건종료=true → 🗑️ 유령연결:임차인(정리필요)
     - 페이지 존재 + 사건종료=false → 🐛 점검버그:쿼리누락 (Wave 1 재쿼리 필요)
     - 페이지 404 / 존재 안 함 → 🚨 유령링크:페이지삭제 (즉시 조사)
4. 이사예정 relation + 공실검증 relation 둘 다 있음 AND 둘 다 유령 아님 → 🚨 이사예정+공실검증 동시연결
5. 아이리스상태=일반임대 가능 AND 공실검증 임대가능여부=일반임대 가능 AND 입력일자 있음 → 🚨 입력일자삭제필요 (일반임대 가능 상태에서 퇴거지정날짜 잔존)
6. (기간내퇴거 or 퇴거날짜지정) AND 거주상태=추가연장X AND 입력일자 < 오늘 → 🚨 퇴실점검필요(날짜경과)
7. **퇴거날짜지정** AND 이사예정 relation 빈값 AND 임차인 relation 있음(유령 아님) → 🚨 이사예정 미등록(퇴거날짜지정)  ← case C (v3.7: 기간내퇴거 제외)
8. (기간내퇴거 or 퇴거날짜지정) AND (거주중 or 추가연장X) AND 엑셀 있음 AND 엑셀≠아이리스상태 → 🚨 엑셀불일치(이사예정)
9. (기간내퇴거 or 퇴거날짜지정) AND (거주중 or 추가연장X) → ✅ 일치:이사예정
10. 임대완료 AND 공실검증=임대완료 AND 엑셀 있음 AND 엑셀≠아이리스상태 → 🚨 엑셀불일치(임대완료)
11. 임대완료 AND 공실검증=임대완료 → ⚠️ 임대완료
12. 공실검증 일치 AND 임차인마스터=특별업무수행 AND 엑셀 있음 AND 엑셀≠아이리스상태 → 🚨 엑셀불일치(특별업무)
13. 공실검증 일치 AND 임차인마스터=특별업무수행 → ✅ 일치:특별업무수행
14. 공실검증 일치 AND 엑셀 있음 AND 엑셀≠아이리스상태 → 🚨 엑셀불일치(공실검증)
15. 공실검증 일치 → ✅ 일치:공실검증
16. 위 전부 해당 없음 → 🚨 아이리스↔해밀시아 상태불일치
```

**🚨 case C — 이사예정 미등록(퇴거날짜지정) 판정 기준** (v3.7 수정):
- 아이리스가 **퇴거날짜지정** 상태인데 이사예정관리 DB에 해당 건이 등록 안 됨
- 의미: 아이리스에선 퇴거날짜지정 입력했는데 이사예정 등록 누락 (역방향 누락)
- 임차인 relation이 있어야 판정 (유령 relation은 rule 3에서 먼저 걸러짐)
- 조치: 윤실장에게 이사예정관리 DB 등록 요청

**🛡️ 기간내퇴거 예외 (v3.7, 2026-04-14 대표님 확정)**:
- **기간내퇴거**는 임차인이 직접 중도해지 통보 → 이사예정관리 건너뛰고 바로 공실검증/아이리스로 가는 업무 플로우
- 따라서 아이리스상태=기간내퇴거 + 이사예정 빈값은 **정상**, 오류 아님
- rule 7은 **퇴거날짜지정에만 적용** (기간내퇴거 제외)
- 기간내퇴거의 정상 흐름: `기간내퇴거 + 공실검증 연결 + 임차인 연결` → ✅ 정상

**🗑️ 유령연결 판정 기준** (v3.3 보강):
- 연결된 페이지가 **완료/종료 상태**인데 relation만 남은 경우
- 공실검증 `확인=true` / 이사예정 `완료=true` / 임차인 `사건종료=true`
- 실제 업무 오류 아님 → 단순 데이터 정리 필요 (relation 제거만 하면 됨)
- 🚨 (업무 오류)와 구분하여 점검보고서에 별도 섹션으로 표시 권장

**🚨 유령링크:페이지삭제 판정 기준** (v3.3 신규):
- 임차인 relation URL이 Wave 1 쿼리 결과(사건종료=false)에 없음 → 개별 `notion-fetch` 수행
- fetch 결과 **404 / 페이지 없음** → 🚨 유령링크:페이지삭제
- 원인: 임차인마스터 페이지가 삭제됐는데 아이리스 페이지에 relation만 고아로 남음 (데이터 꼬임)
- 조치: **즉시 조사 필요** — 어떤 경로로 페이지가 삭제됐는지 + 아이리스 relation 정리
- 점검보고서 DB 등록 시 긴급도=**긴급**, 오류유형=`유령링크`, 담당=대표님

**🐛 점검버그:쿼리누락 판정 기준** (v3.3 신규):
- fetch 결과 페이지 존재 + 사건종료=false
- 원인: Wave 1 임차인마스터 쿼리가 페이지 일부 누락 (페이지네이션/필터 버그)
- 조치: 판정 즉시 중단 → Wave 1 재쿼리 → 재판정
- 점검 시스템 자체 버그 신호

**🚨 임차인중복연결 판정 기준** (신규):
- 아이리스공실 페이지의 1️⃣임차인마스터 relation이 2개 이상
- 원인: 과거 임차인(사건종료됨) + 현재 임차인이 둘 다 연결되어 있음
- 조치: 과거 임차인 relation 제거, 현재 임차인 1개만 남기기

## 📋아이리스[엑셀] vs 아이리스상태👤 자동 비교 (v2.0 Phase 4 추가)

> **🔄최신화검증 formula가 API에서 읽히지 않으므로, 직접 비교한다.**

**비교 전 문자열 정규화** (C2 해결):
- 양쪽 값 비교 전 반드시 정규화 적용: `strip()` + 연속공백→단일공백 + 이모지 제거
- 예: "📑예약중 " → "예약중", " 일반임대  가능" → "일반임대 가능"
- Notion select 필드는 정의된 옵션만 반환하므로 실제 변형은 드물지만, 방어적으로 적용

**정방향 비교** (DB 레코드 기준 — 자동 실행):
아이리스공실 DB 쿼리 시 `📋아이리스[엑셀]` + `아이리스상태👤` 두 필드를 함께 가져와 비교.

| 📋아이리스[엑셀] | 아이리스상태👤 | 판정 | 조치 |
|-----------------|--------------|------|------|
