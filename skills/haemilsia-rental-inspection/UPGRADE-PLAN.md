# 해밀시아 임대점검 스킬 업그레이드 계획

**작성일**: 2026-04-09
**현재 버전**: v1.0 (수동 점검 + 슬랙 알림 + 점검보고서)
**목표 버전**: v2.0 (자동화 + 정확도 향상 + 운영 효율)

---

## 현재 상태 분석

### 잘 되는 것
- 6개 DB 쿼리 + 규칙 판정 → 슬랙 알림 ✅
- 점검보고서 DB 자동 등록 + 소스페이지URL ✅
- 아이리스 엑셀 이미지 → Notion 업데이트 ✅
- 윤실장 2단계 처리 프로세스 ✅

### 문제점
1. 임차인마스터 12규칙 중 rollup 의존 규칙 미점검 (규칙 2,5,6,9,10,11,12)
2. 같은 오류를 매일 중복 등록할 수 있음
3. 슬랙 알림 → 점검보고서 → URL 연결이 3단계로 분리
4. 아이리스 엑셀 vs DB 상태 불일치 자동 검출 없음
5. 전일 미완료 항목 추적 없음

## 업그레이드 로드맵 (5단계)

### Phase 1: 점검보고서 중복 방지
**우선순위**: 높음 | **난이도**: 낮음 | **예상 작업**: 스킬 규칙 추가

**문제**: 점검 실행할 때마다 같은 FLAT2_203 오류가 새 레코드로 등록됨

**해결책**:
1. 점검보고서 DB 쿼리 (☑️처리완료=false) → 미완료 항목 목록 가져오기
2. 새 오류 발견 시 → 기존 미완료 항목과 비교 (비교키로 매칭)
3. 이미 등록된 오류면 → 스킵 + "기존 항목 유지" 로그
4. 새 오류면 → 점검보고서에 등록

**비교키 정규화 규칙**:
- 비교키 = `{건물}_{호수}_{오류유형}` (3필드 복합키)
- 건물명 정규화: 공백 제거, 대소문자 통일 (예: "FLAT 2" → "FLAT2")
- 호수 정규화: 숫자만 추출 (예: "203호", "203", "#203" → "203")
- 오류유형: SKILL.md에 정의된 7개 유형 중 하나 (미입력/불일치/날짜경과/미납/미연결/상태오류/유령링크)
- 예시: `FLAT2_203_미연결` = `FLAT 2_203호_미연결` → 동일 건으로 판정

**완료 기준**: 동일 점검 2회 실행 시 중복 레코드 0건 + 비교키 변형(공백/호 등)에도 정확 매칭

**의존성**: 없음 (즉시 시작 가능)

### Phase 2: 임차인마스터 12규칙 자동화
**우선순위**: 높음 | **난이도**: 높음 | **예상 작업**: 크로스 DB 쿼리 로직

**문제**: rollup 필드(이사예정건수, 공실검증건수 등)가 API에서 omitted 반환

**해결책**:
위성 DB의 relation을 역방향 카운트하여 rollup을 대체.

1. 임차인마스터 전체 레코드 (사건종료=false) 쿼리
2. 각 위성 DB에서 해당 임차인마스터 페이지를 참조하는 레코드 카운트:
   - 이사예정관리: 3️⃣이사예정관리 relation → 임차인마스터 URL 매칭
   - 공실검증: 4️⃣공실검증 relation → URL 매칭
   - 아이리스공실: 6️⃣아이리스공실 relation → URL 매칭
   - 퇴거정산서: 7️⃣퇴거정산서 relation → URL 매칭
   - 미납리스크: 2️⃣미납리스크 relation → URL 매칭
3. 카운트 결과로 12규칙 판정:
   - 규칙 1: 이름/건물/호수 비어있음 → 직접 필드 체크 (이미 가능)
   - 규칙 2: 공실 이중 등록 → 공실검증 건수 vs 아이리스공실 건수 비교
   - 규칙 5: 이사예정 완료 + 퇴거정산서 없음 → 이사예정 건수>0 + 퇴거정산서 건수=0
   - 규칙 9: 거주중+조건부퇴거 + 아이리스미등록
   - 규칙 11: 이사예정+공실검증 동시 연결 (복합 분기)
   - 규칙 12: 신규입주 불필요링크 → 이사예정=0 + 공실검증=0 + 다른 링크 존재

**구현 접근**:
- 방법A: 모든 위성 DB를 쿼리 → 메모리에서 크로스 매칭 (정확하지만 느림)
- 방법B: 임차인마스터 레코드의 relation 필드만 확인 (빈값 vs 값있음) → 건수 대신 유무 판정 (빠르지만 일부 규칙 제약)
- **추천: 방법B 우선 → 빈값/유무로 판정 가능한 규칙부터 구현, 건수 필요한 규칙은 Phase 2.5로**

**미지원 규칙 목록 + 사유**:
| 규칙 | 내용 | 지원 여부 | 사유 |
|------|------|----------|------|
| 규칙 1 | 이름/건물/호수 비어있음 | ✅ 지원 | 직접 필드 체크 |
| 규칙 2 | 공실 이중 등록 | ✅ 지원 | relation 유무로 판정 가능 |
| 규칙 3 | 거주상태 미입력 | ✅ 지원 | 직접 필드 체크 |
| 규칙 4 | 만기일 미입력 | ✅ 지원 | 직접 필드 체크 |
| 규칙 5 | 이사예정 완료 + 퇴거정산서 없음 | ✅ 지원 | relation 유무로 판정 |
| 규칙 6 | 퇴거정산서 있음 + 이사예정 없음 | ✅ 지원 | relation 유무로 판정 |
| 규칙 7 | 사건종료 + 미해결 위성 링크 | ⚠️ Phase 2.5 | 모든 위성 DB 카운트 필요 |
| 규칙 8 | 임대료 미입력 | ✅ 지원 | 직접 필드 체크 |
| 규칙 9 | 거주중+조건부퇴거 + 아이리스미등록 | ✅ 지원 | relation 유무로 판정 |
| 규칙 10 | 미납 중복 등록 | ⚠️ Phase 2.5 | relation 건수 카운트 필요 |
| 규칙 11 | 이사예정+공실검증 동시 연결 | ✅ 지원 | relation 유무로 판정 |
| 규칙 12 | 신규입주 불필요링크 | ⚠️ Phase 2.5 | 복합 건수 비교 필요 |

> Phase 2에서 9개 규칙 지원, Phase 2.5에서 나머지 3개(규칙 7,10,12) 추가 예정

**Ghost Link (삭제된 페이지 참조) 필터 처리**:
- relation 필드에 값이 있어도 참조 대상이 삭제된 경우 존재
- 구현: relation URL로 `notion-fetch` 시도 → 404/에러 반환 시 Ghost Link로 판정
- Ghost Link 발견 시 → 점검보고서에 "유령링크" 오류유형으로 등록
- 성능 고려: 전체 relation을 검증하면 API 호출 과다 → **오류 판정된 건만 Ghost Link 검증** (정상 건은 스킵)

**Notion API pagination 처리**:
- Notion API는 한 번에 최대 100건 반환 → `has_more=true` 시 `start_cursor`로 다음 페이지 요청
- 임차인마스터 (사건종료=false) 현재 ~50건 → 단일 페이지로 충분
- 미납리스크/아이리스공실 등 100건 초과 가능성 → **모든 DB 쿼리에 pagination 루프 적용**
- 구현: `while has_more: fetch(start_cursor) → append results → update cursor`

**완료 기준**: 12규칙 중 최소 9개 자동 판정 + 점검보고서 자동 등록 + Ghost Link 0건 누락

**의존성**: Phase 1 완료 후 (중복 방지 필요)

### Phase 3: 슬랙+점검보고서 동시 생성
**우선순위**: 중간 | **난이도**: 낮음 | **예상 작업**: 워크플로우 통합

**문제**: 슬랙 알림 → 점검보고서 → URL 연결이 각각 별도 단계

**해결책**:
1. 점검 로직 실행 → 오류 목록 생성
2. 오류 목록으로 동시에:
   a. 슬랙 메시지 조합 + 전송
   b. 점검보고서 DB에 등록 (소스페이지URL 포함)
   c. 슬랙 메시지에 점검보고서 링크 포함
3. 스킬의 실행 순서를 "점검→판정→{슬랙+보고서 동시}"로 변경

**부분 실패 시 재시도 전략**:
- 슬랙 전송 실패 + DB 등록 성공 → 슬랙만 재시도 (최대 3회, 간격 5초)
- 슬랙 전송 성공 + DB 등록 실패 → DB만 재시도 (최대 3회, 간격 5초)
- 둘 다 실패 → Notion 점검보고서 DB에 `처리상태="전송실패"` 레코드 등록 (Railway `/tmp`는 재시작 시 유실되므로 사용 금지)
- 3회 재시도 후에도 실패 → 슬랙 #haemilsia-task 채널에 "⚠️ 점검 알림 전송 실패" 에러 메시지 별도 전송 (가능한 경우)
- 다음 점검 실행 시 `처리상태="전송실패"` 레코드 조회 → 미전송 건 재시도
- **원칙: 한쪽 성공은 보존하고, 실패한 쪽만 재시도**

**완료 기준**: 점검 1회 실행 시 슬랙+보고서가 동시에 완료 + 부분 실패 시 정상 복구

**의존성**: Phase 1 완료 후 (중복 방지 적용 상태에서)

### Phase 4: 아이리스 엑셀 자동 비교
**우선순위**: 중간 | **난이도**: 중간 | **예상 작업**: 비교 로직 + 점검보고서 연동

**문제**: 📋아이리스[엑셀] vs 아이리스상태👤 불일치를 수동으로만 확인

**해결책**:
1. 아이리스공실 DB 쿼리 시 📋아이리스[엑셀] + 아이리스상태👤 함께 가져오기
2. 두 필드 비교:
   - 동일 → 정상
   - 불일치 → 점검보고서에 "상태불일치" 오류 등록
   - 📋아이리스[엑셀] 빈값 → "엑셀 미입력" 참고 등록
3. 🔄최신화검증 formula가 이미 이 비교를 하지만 API에서 못 읽으므로, 직접 비교

**예외 처리**:
- 엑셀 업데이트 직후라 아이리스상태가 아직 안 바뀐 경우 → 경고만 (에러 아님)
- 둘 다 빈값 → 별도 "미입력" 플래그

**양방향 비교**:
- **정방향**: DB 레코드 기준 → 📋아이리스[엑셀] ≠ 아이리스상태👤 불일치 검출
- **역방향**: 엑셀 기준 → 엑셀에는 있는데 DB에 레코드가 없는 호실 검출
  - 구현: 엑셀 이미지에서 추출한 호실 목록 vs 아이리스공실 DB 전체 레코드의 건물+호수 비교
  - 엑셀에만 있는 호실 → "DB 미등록" 경고 (점검보고서에 "참고" 등급으로 등록)
  - DB에만 있는 호실 → "엑셀 미반영" 경고 (엑셀 업데이트 필요)

**완료 기준**: 📋아이리스[엑셀] ≠ 아이리스상태👤 불일치 자동 검출 + 엑셀↔DB 양방향 누락 검출

**의존성**: Phase 1 완료 후

### Phase 5: 미해결 항목 추적 (슬랙 리마인더)
**우선순위**: 낮음 | **난이도**: 낮음 | **예상 작업**: 슬랙 추가 메시지

**문제**: 전일 등록 오류가 아직 미완료인지 추적 안 됨

**해결책**:
1. 점검 시작 시 점검보고서 DB에서 ☑️처리완료=false 항목 조회
2. 등록 후 N일 경과한 미완료 항목 필터
3. 슬랙 알림에 "미해결 항목" 섹션 추가:
   ```
   ━━━ 📌 미해결 항목 (N건) ━━━
   ⏰ [3일 경과] FLAT2_203 공실검증 미연결 — 담당: 이대리
   ⏰ [1일 경과] FLAT2_301 상태미입력 — 담당: 이대리
   ```

**에스컬레이션 경로**:
| 경과일 | 등급 | 조치 |
|--------|------|------|
| 1일 | 알림 | 슬랙 미해결 섹션에 표시 (기본) |
| 3일 | 리마인더 | 담당자 멘션 + "3일 경과" 강조 |
| 7일 | 에스컬레이션 | 대표님 멘션 + 긴급도 자동 승격 (참고→확인, 확인→긴급) |
| 14일 | 최종 경고 | 대표님 DM + 점검보고서에 "장기미해결" 태그 추가 |

- 담당자별 멘션 ID: 윤실장/이대리는 슬랙 메시지 내 이름 표기, 대표님은 별도 DM
- 처리완료 체크 시 → 에스컬레이션 타이머 자동 해제

**완료 기준**: 미완료 2일+ 항목이 슬랙에 자동 표시 + 경과일별 에스컬레이션 자동 실행

**의존성**: Phase 1 + Phase 3 완료 후

## 실행 순서 요약

```
Phase 1 (중복방지) ─────────┐
                             ├─→ Phase 2 (12규칙)
                             ├─→ Phase 3 (동시생성)
                             ├─→ Phase 4 (엑셀비교)
                             └─→ Phase 5 (미해결추적)
```

**Phase 1이 모든 후속 Phase의 전제조건.**
Phase 2~4는 Phase 1 완료 후 병렬 진행 가능.
Phase 5는 Phase 3 이후.

## API Rate Limit 핸들링 (전체 Phase 공통)

모든 Notion/슬랙 API 호출에 적용하는 공통 규칙:

| API | Rate Limit | 핸들링 |
|-----|-----------|--------|
| Notion API | 3 req/sec | 호출 간 `asyncio.sleep(0.35)` 삽입 |
| Slack API | 1 req/sec | 호출 간 `asyncio.sleep(1.1)` 삽입 |

- 429 응답 시 → `retry-after` 헤더 값만큼 대기 후 재시도 (최대 3회)
- Ghost Link 검증 등 대량 호출 예상 구간 → 배치 처리 (10건 단위 + 3초 간격)
- 전체 점검 실행 예상 API 호출: ~50~100회 → 약 30초~1분 소요 (정상 범위)
