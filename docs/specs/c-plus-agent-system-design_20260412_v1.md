# C+ 에이전트 시스템 설계 (Agent System C+ Design)

**버전**: v1.0
**작성**: 2026-04-12
**상태**: 브레인스토밍 완료 → 대표님 리뷰 대기
**원작자**: 이현우 대표님 (아이디어 + 방향 결정) + Claude (설계 + 문서화)
**프로젝트명**: C+ (C안 + 모델별 분배)

---

## 0. 요약

해밀시아 Claude 운영 시스템을 **"1인 만능 매니저"에서 "1인 오케스트레이터 + 18명 전문 팀"으로 전환**하는 프로젝트.

기존 agent.md(v1.1, 종이호랑이)를 재탄생시키고, 세션 시작/종료/MODE 루틴을 **병렬 서브에이전트 디스패치** 구조로 재작성. 팀원은 Claude 모델 3등급(Haiku/Sonnet/Opus)으로 작업 특성별 분배.

### 핵심 기대 효과

| 지표 | 지금 | C+ 후 |
|------|------|-------|
| 세션 시작 | 45~60초 | 10초 (6배) |
| 세션 종료 | 3~5분 | 15초 (15~20배) |
| MODE 1 기획 사이클 | 4분 | 2분 (2배) |
| 에러 자동 복구율 | ~20% | 80% |
| 월 Opus 비용 | 100% | 50% (절반 절감) |
| B2 위반 (세션 인수인계 누락) | 48회 누적 | 0회/월 (시스템 강제) |
| B8 위반 (통합본 누락) | 34회 누적 | 0회/월 (시스템 강제) |
| 긴 세션 유지력 | 2시간 | 4~6시간 |
| 학습 카드 품질 | Sonnet (중) | Opus (최고) |

## 1. 배경

### 1.1 발단

2026-04-12 대표님이 agent.md(v1.1, 2026-04-05) 점검 중 본질적 문제 발견:

> "이게 이젠 별 필요가없는거같아! 다 중복이고 의미가 없어!! 에이전트를 활용해야겠어!
> 서브에이전트 팀에이전트 지금 세션마무리할때도 오래걸리고 세션시작할때도 오래걸리고
> 작업하나를 하면 시간이 너무걸려!"

### 1.2 문제 진단

1. **agent.md v1.1 종이호랑이화**: plan-agent + 서브플래너 4명이 설계만 존재, 실제 Task 툴 기반 구현 0개. CLAUDE.md MODE 1 워크플로우와 90% 중복.
2. **순차 병목**: 세션 시작/종료/MODE 전이가 모든 단계 Opus 순차 실행. 독립 작업인데 병렬화 없음.
3. **모델 낭비**: 단순 Notion 쿼리도 Opus 4.6 사용. 비용 과다 + 속도 저하.
4. **47개 서브에이전트 활용 0%**: Task 툴의 subagent_type(Explore, Plan, gsd-*, superpowers:*) 미활용.
5. **B2/B8 누적 위반**: 세션 인수인계 미생성 48회, Git→Notion 통합본 누락 34회. 시스템 구조가 실수 허용.

### 1.3 기회

- Claude Code Task 툴 + model 파라미터 이미 제공
- superpowers:dispatching-parallel-agents 스킬 이미 설치
- 추가 인프라 0: 기존 도구만으로 구현 가능

## 2. 목표

### 2.1 주요 목표

1. **agent.md 재탄생**: 종이호랑이 → 19명 진짜 팀원 레지스트리
2. **병렬 기본화**: 독립 작업은 무조건 병렬 dispatch
3. **모델 분배**: Haiku/Sonnet/Opus를 작업 특성별 자동 선택
4. **에스컬레이션 자동화**: 실패 시 수동 개입 없이 자동 승급
5. **누적 위반 0화**: B2/B8 재발 방지를 시스템 구조로 강제

### 2.2 부차 목표

- 메인 context 부하 감소 → 긴 세션 유지력 2~3배 증가
- 월 Opus 비용 절감 → 대표님 일일 사용 한도 확장
- 학습 루프 품질 상승 (복습카드관 Opus 승급)
- 대표님 즐거움/몰입도 관리 (분위기메이커 신설)
- 다른 팀원들이 깨끗한 환경에서 일할 수 있게 유지 (경비원/청소원 신설)
- C+ 시스템 자체의 정기 감사 (외주 감사관 신설 — 2주 후 첫 감사, 이후 월 1회)

## 3. 비목표 (Non-goals)

이 프로젝트가 건드리지 않는 것:

- 메인 세션 모델 격하 — 매니저(대표님 대화 전담)는 Opus 4.6 유지, 절대 격하 금지
- 기존 스킬 삭제 — 모든 스킬 유지. 에이전트가 스킬을 래핑만.
- 기존 REF 훅 제거 — 훅 25개 유지. 경비원이 상위 layer로 감쌈.
- 메모리 파일 대량 삭제 — 기존 메모리 전부 유지. 청소원은 아카이브만.
- CLAUDE.md 전면 재작성 — 소폭 업데이트만. MODE 1~4 구조 유지.
- 대화 UX 변경 — 대표님 입장에선 대화 흐름 동일. 내부만 병렬화.
- 에이전트 자율 결정 권한 확대 — 에이전트는 매니저 지시만 실행.

| 용어 | 정의 |

## 5. 아키텍처

### 5.1 전체 구조

```
                    ┌─────────────────────────┐
                    │      대표님 (사장님)      │
                    └────────────┬────────────┘
                                 │
                                 ▼
        ┌──────────────────────────────────────────────┐
        │         총괄매니저 = Claude (Opus 4.6)         │
        │  ──────────────────────────────────────────  │
        │  - 대표님 대화 독점 (Opus 고정, 격하 금지)       │
        │  - 의도 해석 → 트리거 매칭 → dispatch 결정      │
        │  - 팀원 결과 병합 → 대표님께 1줄 보고           │
        │  - 수동 오버라이드 키워드 수신                 │
        └──────────────┬───────────────────────────────┘
                       │ Task 툴 병렬 dispatch
         ┌─────────────┼─────────────┬─────────────┐
         ▼             ▼             ▼             ▼
    ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐
    │ 팀원 A  │  │ 팀원 B  │  │ 팀원 C  │  │ 팀원 D  │
    │ (모델별) │  │ (모델별) │  │ (모델별) │  │ (모델별) │
    └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘
         └─────────────┴─────────────┴─────────────┘
                       │ 결과 반환
                  매니저 병합 → 대표님 보고
```

### 5.2 4대 기둥

1. **총괄매니저 (Opus 4.6, 불변)** — 대화/라우팅/병합 전담. 직접 실행하는 건 대화, 병합, 라우팅 판단뿐.
2. **팀원 풀 (19명, 모델 등급별)** — Haiku 7명(인턴) + Sonnet 5명(팀장) + Opus 7명(임원).
3. **혼합 라우팅 엔진** — 자동 트리거 테이블(기본) + 수동 오버라이드(`/agent [이름]`) + 수동 시 매니저 추천 의무.
4. **에스컬레이션 체인** — Haiku→Sonnet→Opus 자동 승급. Opus 실패 시에만 대표님 개입.

### 5.3 3개 레이어

```
┌──────────────────────────────────────────────┐
│  Layer 3: 전략/창의 (임원 Opus 6명)             │
│  기획/아이디어/CEO/ENG/복습카드/분위기          │
├──────────────────────────────────────────────┤
│  Layer 2: 실무/판단 (팀장 Sonnet 5명)           │
│  핸드오프/코드리뷰/QA/Preflight/청소            │
├──────────────────────────────────────────────┤
│  Layer 1: 실행/감시 (인턴 Haiku 7명)             │
│  Notion/Slack/파일/경비/메모리/지침/도구        │
└──────────────────────────────────────────────┘
       ↑ 총괄매니저 (Opus) ↑ 대표님
```

### 5.4 수동 오버라이드 시 매니저 행동

| 시나리오 | 매니저 행동 |
|---------|-----------|
| 대표님이 애매하게 "수동으로 해줘" | 추천 1~3개 + 이유 + 모델등급 + 예상 소요 제시 |
| 대표님이 명시적 `/agent [이름]` | 무조건 실행 + soft 대안 제안 1줄 |
| 대표님이 "뭐 써야 돼?" | 상황 분석 + 추천 3개 (1순위/2순위/3순위) |

추천 원칙: 항상 1~3개, 각 한 줄 이유, 모델 등급 + 예상 소요 명시, 병렬 가능 조합 묶어서 제안.

### 6.2 모델 비중

- Haiku (인턴): 7명 — 비용 ~15%
- Sonnet (팀장): 5명 — 비용 ~30%
- Opus (임원): 7명 — 비용 ~50%
- 매니저 (Opus): 1명 — 비용 ~5%

### 6.3 설계 원칙

- 복습카드관은 Opus 고정 (대표님 직접 결정: "실력향상에 중요하거든!")
- 학습/복습 관련 에이전트는 비용 절감 대상 아님, 무조건 Opus
- 분위기메이커 억지 유머 금지 — 진지 모드에선 침묵. 적시성 > 빈도. 자동 최소 간격 20분, 하루 최대 5회
- 외주 감사관은 매니저와 완전 독립 — 매니저도 감사 대상. 결과 수정 불가, 대표님에게 직접 보고. P7 완료 2주 후 첫 감사, 이후 월 1회 정기
- 경비원: 기존 REF 훅 삭제 안 함, 상위 해석 layer로 감쌈
- 청소원: 삭제 금지 기본값, 이동만. 실제 삭제는 대표님 승인 후에만

### 6.4 각 팀원 상세 프로필

각 에이전트의 트리거 / 입력 / 출력 / 도구셋 / 예상 소요 / 특수 규칙은 `~/.claude/agents/{agent-id}.md` 파일에 개별 정의. agent.md v2.0은 요약 표 + agents/ 디렉토리 참조.

## 7. 업무 흐름

### 7.1 세션 시작 (지금 60초 → C+ 10초)

지금: Opus 순차로 TOP 5 → MEMORY → rules → skill-guide → 인사 (45~60초)

C+:
```
t=0s   대표님 "세션시작"
t=0.2s 매니저가 4~5명 동시 dispatch:
       [규칙감시관 Haiku] [기억관리관 Haiku] [지침사서 Haiku]
       [분위기메이커 Opus] + [청소원 Sonnet] (매일 첫 세션만)
t=7s   전원 완료 → 매니저 병합
t=8s   통합 응답 출력 (TOP 5 + 메모리 + 환경(해당시) + 환영 + 인사)
```
> 규칙감시관 Notion 지연 시: 에스컬레이션 없이 1회 타임아웃 → 즉시 폴백 (캐시 참조)

### 7.2 세션 종료 (지금 3~5분 → C+ 15초)

C+:
```
t=0s   대표님 "마무리"
t=0.3s Stage 1: 5~6명 동시 dispatch
       [규칙감시관] [노션기록관] [노션기록관(에러로그)] [핸드오프작성관] [복습카드관] + [청소원](가벼운 청소)
t=10s  Stage 1 완료 → 매니저 병합
t=11s  Stage 2: [슬랙배달관] 발송 (Stage 1 결과 필요)
t=14s  최종 보고
```

### 7.3 MODE 1 기획 사이클 (지금 4분 → C+ 2분)

| 단계 | C+ 방식 | 소요 |
|------|--------|------|
| 맥락 수집 | 기억관리관+지침사서+도구추천관 Haiku 병렬 | 5초 |
| 아이디어 검증 | 아이디어검증관 Opus | 30초 |
| 브레인스토밍 | 매니저 직접 (대화) | 수 분 |
| CEO + ENG 리뷰 | 두 Opus 리뷰어 **병렬** | 25초 |
| writing-plans | 기획플래너 Opus | 30초 |
| Preflight 검증 | 3인조 **병렬** | 15초 |
| 이해 브리핑 | 매니저 직접 | 30초 |

### 7.4 에러 에스컬레이션

```
[Haiku 실패] → 자동 [Sonnet 재시도] → 자동 [Opus 재시도] → 대표님 보고
모든 단계에서 에러로그 DB 자동 기록
```

### 7.5 3대 운영 원칙

1. **병렬이 가능하면 무조건 병렬** — 순차는 데이터 의존이 있을 때만
2. **실패는 자동 승급** — 대표님 개입은 Opus까지 실패 후에만
3. **매니저는 조합만, 실행은 팀원** — 매니저가 직접 하는 건 대화/병합/라우팅뿐
4. **Notion 읽기 실패는 에스컬레이션 안 함** — 1회 타임아웃 시 즉시 폴백 반환 (캐시 참조). 병렬 전체 블록 방지. 쓰기 실패만 에스컬레이션 대상.
5. **대표님 대기시간 최소화** — 병렬 결과 중 1명이 지연되면, 매니저는 "나머지 N명 결과 먼저 + 1명 조회 중" 으로 부분 응답 가능

## 8. 에러 처리

### 8.1 에러 분류 6종

| # | 유형 | 자동 대응 |
|---|------|---------|
| 1 | 팀원 실행 실패 | Haiku→Sonnet→Opus 자동 승급 |
| 2 | 타임아웃 (예상 x3배 초과) | 강제 중단 + 같은 등급 재시도 → 승급 |
| 3 | 외부 서비스 장애 (Notion/Slack) | 로컬 큐잉 `~/.claude/queue/pending_*.json` → 복구 후 재전송 |
| 4 | 팀원 의견 충돌 | 매니저가 양쪽 의견 구조화 보고 + 중립 추천 → 대표님 판단 |
| 5 | 결과 오염 | sanity check 3단계 (포맷/범위/일관성) → 재dispatch |
| 6 | 대표님 취소 (Ctrl-C) | graceful shutdown + 부분 결과 저장 + "이어서 계속?" 질문 |

### 8.2 타임아웃 한도

| 등급 | 평균 소요 | 타임아웃 한도 |
|------|---------|------------|
| Haiku | 3초 | 10초 |
| Sonnet | 8초 | 25초 |
| Opus | 15초 | 45초 |

### 8.3 자동 복구 한도

- 같은 팀원 재dispatch 최대 2회
- 에스컬레이션 최대 3 tier
- 알려진 버그: 메모리 파일 참조 → 우회 전략 자동 적용 (예: Notion MCP 파싱 버그 → update_content)
- 같은 에러 2회 → 메모리 파일 승격 제안, 3회 → 경비원 PreToolUse 자동 차단

### 8.4 매니저 sanity check 3단계

1. **포맷 검증**: 예상 스키마와 일치 여부 (행 수, 필수 필드)
2. **범위 검증**: 값이 상식 범위 내인지
3. **일관성 검증**: 다른 팀원 결과와 모순 없는지

## 9. 검증 전략

### 9.1 Before/After 벤치마크

Phase 0에서 기준선 5회 측정 → 각 Phase 완료 시 비교.
저장 위치: `~/.claude/benchmarks/`

### 9.2 성공 지표 + 합격선

| 지표 | 지금 | 합격선 | 목표 |
|------|------|-------|------|
| 세션 시작 | 45~60초 | 20초 이하 | 10초 |
| 세션 종료 | 3~5분 | 45초 이하 | 15초 |
| MODE 1 사이클 | 4분 | 3분 이하 | 2분 |
| Opus 일일 토큰 | 100% | 70% 이하 | 50% |
| B2 위반 신규 | - | 0회/월 | 0회/월 |
| B8 위반 신규 | - | 0회/월 | 0회/월 |
| 에러 자동 복구율 | ~20% | 60% 이상 | 80% |
| 대표님 주관 만족도 | - | 4/5 이상 | 5/5 |
| 회귀 테스트 통과율 | - | 100% | 100% |

1개월 후: 7개 이상 달성 → C+ 완전 정착 / 5~6개 → 조정 후 재평가 / 4개 이하 → 롤백 검토

### 9.3 회귀 테스트

저장 위치: `~/.claude/tests/c-plus-regression/`
자동 실행: 주 1회 (훅)
테스트 케이스: TOP 5 결과 일치, handoff 스타일 유사, 복습 카드 구조 준수, Preflight 점수 ±5% 오차

### 9.4 Dry-run 모드

`agent.md`에 `mode: dry-run` 설정 시:
- 모든 dispatch 정상 실행
- Notion/Slack 쓰기 → intercept → 콘솔에 "이럴 거였음" 출력
- 외부 상태 변화 0
- Phase 1 직후 1일 필수 적용

### 9.5 Gradual Rollout

P0~P7 단계적 도입 (섹션 10 참조). 각 Phase 게이트: 합격선 충족 + 대표님 주관 평가 + 에러 로그 리뷰.

### 9.6 롤백 전략

- Git 브랜치 per Phase
- 에이전트별 `enabled: true/false` 스위치
- 점진적 롤백 (문제 팀원만 비활성화)
- 최종 롤백: `git revert` + agent.md 비활성화 (메모리/학습 카드는 유지)

## 11. 리스크 + 대안

### 11.1 주요 리스크

| 리스크 | 확률 | 영향 | 대응 |
|-------|-----|-----|------|
| 병렬 dispatch 오버헤드가 이득 상쇄 | 낮 | 중 | P2 벤치마크 실측 → 미달 시 조정 |
| 에이전트 결과 품질 저하 | 중 | 높 | sanity check 3단계 + Dry-run 필수 |
| Opus 절감 기대 미달 | 중 | 낮 | 보수적 목표 30%로 설정 |
| 대표님 대화 UX 변화 | 낮 | 중 | 매니저 Opus 불변, 대화 흐름 동일 |
| 마이그레이션 도중 시스템 마비 | 낮 | 높 | Phase별 Git + 즉시 롤백 |

### 11.2 검토했으나 채택 안 한 대안

| 대안 | 기각 이유 |
|------|---------|
| GSD 기존 에이전트 풀만 활용 | 해밀시아 운영 특화 작업 미지원 (한국어 지침/비유/Notion DB) |
| 단일 마스터 오케스트레이터 에이전트 | 추가 layer = latency 증가. 매니저 직접 dispatch가 더 빠름 |
| 외부 workflow engine (Temporal 등) | 인프라 과도. Task 툴로 충분. YAGNI |
| 점진적 5명→10명→18명 | 에이전트 간 상호작용 미설계 → 나중에 재설계 비용 발생 |
