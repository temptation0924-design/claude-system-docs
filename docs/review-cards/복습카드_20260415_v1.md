## 카드 1: B18 — Agent dispatch 파일 누락 방지

**Why** — 빡센점검 Wave 2a에서 매니저가 Agent A에게 **임차인+아이리스 2개 파일만 전달**. 나머지 5개 DB JSON 경로 누락 → Agent가 cross-DB URL 매칭을 못 하고 rollup 필드(`<omitted/>`) 의존 → **11분기 판정불가**. 스킬 문서에는 URL 매칭 원칙이 있었으나 실행 프롬프트 즉흥 작성으로 누락.

**How** —
1. `docs/agents.md`에 **Wave 2 SELF-CHECK 4단계** 추가:
   - Wave 1에서 7개 JSON 저장됐는지
   - Agent 프롬프트에 7개 경로 전부 포함했는지
   - 규칙 파일 경로 포함했는지
   - "rollup 의존 금지, URL 매칭 사용" 명시했는지
2. **Agent A/B 표준 프롬프트 템플릿** (복붙용) 블록 추가 — `{TS}` 타임스탬프만 치환.
3. Notion 규칙위반 DB에 **B18 신규 select 옵션** + 초기 row 등록 (반복 1회).
4. `rules.md` B18 훅 현황표 행 추가 (수동 감시, 3회 이상 누적 시 훅 승격 검토).

**Takeaway** — **스킬 문서 ≠ 실행 프롬프트**. 원칙이 문서에 있어도 즉흥 프롬프트 작성 시 놓칠 수 있음. 체크리스트 + 복붙 템플릿으로 **구조적 방지**가 유일한 해법. 검증: 재실행 결과 11건 전부 실제 판정 성공.

## 카드 2: 3단 분류 체계 (A / B / C)

**Why** — 기존 빡센점검이 "오류 41건"으로 집계 → 실제로는 대부분 진행 중 워크플로우. 미납리스크 28건·퇴거정산 진행중 11건을 "오류"에 섞어 넣으면 **진짜 오류가 묻힘** + 담당자 둔감화. "오류=즉시 수정"과 "확인=액션"과 "진행 중=참고"는 담당자 반응이 달라야 하는데 섞여있으면 반응 속도가 떨어짐.

**How** —
```
A: 입력 오류  ❌  — 데이터 누락·미연결·유령링크·거주상태 빈값
B: 확인사항  ✋  — 체크박스·토글·독촉·긴급대처(승격)
C: 진행 추적  📊  — 상습·문자·대기·진행중 (건수만, 상세 숨김)
```

**경계 케이스 3건 확정** (메모리 영구 보관):
- 미납 긴급대처(>30일 위험) → **B로 승격** (C 아님 — 독촉 액션 필요)
- 이사예정 거주상태 빈값 → **A로 강등** (B 아님 — 필수 입력 누락)
- 퇴거정산서 v4.0 결핍(파일/송부일/계좌번호) → **B** (워크플로우 필드)

`docs/slack-format.md` v2.0에 섹션 분리 템플릿, `docs/agents.md`에 `category` 필드 강제.

**Takeaway** — 오류 분류는 **담당자 의사결정 정확도**를 좌우. "A 0건" 달성이 목표, C는 많아도 문제 아님. 애매하면 B로 두면 담당자가 무시할 수 있지만 놓치지 않음.

## 카드 3: 퇴거정산서 v4.0 — M2 마이그레이션

**Why** — rules/6 STUB을 4분기 결핍 판정으로 확장. DB에 필드 3개 추가 필요 (정산서파일·송부일·계좌번호). 문제: **기존 27건은 필드가 비어있어 전부 오류로 잡힘**. 소급 수동 입력은 업무량 크고 정확성 불확실.

**How — M2 전략 (시스템 적용일 컷오프)**:
- 적용일 `2026-04-15` 명시
- 판정 로직: `created_time < 2026-04-15` 레코드는 **점검 제외**
- 신규 생성 건만 결핍 판정 대상

**판정 4분기**:
| 상태 | 결핍 | 알림 |
|------|------|------|
| 정산서결제대기 | 정산서파일 empty | 윤실장: 업로드 필요 |
| 정산서송부 | 송부일 NULL | 윤실장: 송부일 기록 |
| 진행중[입금대기] | 계좌번호 empty | 윤실장: 계좌번호 기재 |
| 진행중[입금대기] | 계좌번호 있음 | 대표님: 입금 필요 |

**Takeaway** — 스키마 변경 시 **마이그레이션 전략(M1 소급 / M2 컷오프 / M3 태그)** 을 **명시적으로 선택**해야 한다. 기본값 없이 시스템 배포하면 기존 데이터가 전부 오류로 폭발. M2는 "신규 건부터 서서히 정비"라 업무 충격 최소.
