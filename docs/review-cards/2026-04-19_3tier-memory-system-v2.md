---

🎯 핵심 개념 (대표님이 오늘 새로 배운 것):

• **3계층 역할 분리 (MEMORY / handoff / Notion)**
  → 해밀시아 운영에 빗대면:
    - **MEMORY.md** = 현관 입구 **화이트보드** (오늘 볼 것만 한눈에, 인덱스)
    - **handoff/** = 서재의 **업무일지 원본** (진짜 기록, 펼쳐 보는 파일)
    - **Notion DB** = 은행 **금고 백업** (불나도 살아남는 복제본)
  → 화이트보드에만 "202호 입주 완료"라고 써놓고 계약서 원본이 없으면? 드리프트 발생.

• **handoff-first 쓰기 순서 (원본 우선, 인덱스는 best-effort)**
  → 입주 처리할 때 **계약서(원본)를 먼저 금고에** 넣고, 화이트보드는 "나중에 써도 OK".
  → 기존 방식: 화이트보드+계약서+금고 한꺼번에 기록 → 중간에 하나 실패하면 drift
  → 새 방식: **계약서(handoff) 저장 성공 = 작업 완료**로 간주. MEMORY/Notion은 실패해도 queue에 던져두고 나중에 재시도.

• **mkdir-lock 패턴 (맥OS에 flock 없어서 우회)**
  → 관리소에 "작업중 팻말"이 하나뿐일 때, 두 직원이 동시에 팻말 걸려다 충돌하는 문제.
  → `mkdir /tmp/lock.d`는 원자적 (이미 있으면 실패) → **폴더 만들기 = 락 획득**으로 대체.
  → bash에서 `flock` 대신 `mkdir` 한 줄로 동시성 제어 가능.

• **자동화 훅 5개 (A~E)**
  → 사람이 안 까먹게 **건물 자체가 알람을 울리게** 만든 것.
    - A: SessionStart 자동 주입 (세션 시작하면 화이트보드 먼저 들이밂)
    - B: 세션 종료 패치 (나갈 때 체크리스트 자동)
    - C: 매일 archive (오래된 메모는 서랍으로 이동)
    - D: 반복 위반 예방 injection (같은 실수 3번 하면 규칙이 먼저 뜸)
    - E: drift 감지 (화이트보드 ↔ 원본 불일치 감지)

• **queue 재시도 패턴**
  → 화이트보드 업데이트 실패해도 **"미처리 큐" JSON**에 던져놓고 다음 세션 시작 때 자동 재시도.
  → 원자적으로 동시 3곳 쓰기 불가능한 환경에서 **실용적 최선책**.

• **CEO / ENG / Preflight 3각 리뷰**
  → 한 계획을 3명이 다른 관점으로 봄:
    - CEO: "이거 할 가치 있어?" (스코프·우선순위)
    - ENG: "기술적으로 말되게 설계됐어?" (아키텍처)
    - Preflight: "실제로 돌아갈까?" (실행 가능성·의존성)
  → 실제로 이번 세션도 3명이 각자 수정 요청 냈고, 그 덕에 drift 감지 훅이 추가됨.

---

🔄 과정 시각화:

**Before (7차 세션까지)**:
```
작업 1건 → [MEMORY + handoff + Notion] 동시 쓰기 시도
              ↓ 하나라도 실패하면
          drift 발생 (실제 사례: notion_synced=false인데 MEMORY엔 "완료")
```

**After (v2.0)**:
```
작업 1건
   ↓
① handoff 저장 (원본, 반드시 성공) ✅ 여기서 작업 "완료" 선언
   ↓
② MEMORY 업데이트 (best-effort) — 실패 시 → queue.json 적재
   ↓
③ Notion 싱크 (best-effort) — 실패 시 → queue.json 적재
   ↓
[다음 세션 시작]
   ↓
SessionStart 훅 → queue.json 검사 → 미처리 건 자동 재시도
   ↓
drift 감지 훅 → MEMORY ↔ handoff 불일치 있으면 경고
```

**3계층 구조**:
```
┌────────────────────────────────────────┐
│ 현관 (MEMORY.md)      — 인덱스, 빠른 조회│  ← 휘발성 OK
├────────────────────────────────────────┤
│ 서재 (handoffs/*.md)  — 원본, 업무일지   │  ← Source of Truth
├────────────────────────────────────────┤
│ 금고 (Notion DB)      — 영구 백업        │  ← 복구용
└────────────────────────────────────────┘
```

---

✨ 잘된 부분:

• **"원본은 뭔가"를 명시적으로 선언** — handoff가 SoT라고 못박았기 때문에 drift 발생 시 복구 방향(handoff → MEMORY/Notion)이 자동으로 결정됨.
• **flock 없이도 lock 구현** — macOS 기본 환경 제약을 mkdir 원자성으로 우회한 현실적 설계.
• **실패를 수용하는 설계** — "3곳 동시 쓰기 성공"을 요구하지 않고 "원본만 성공, 나머지는 이연"으로 내려놓은 판단.
• **3각 리뷰가 실제로 가치 냄** — Preflight가 "drift 감지 훅(E) 빠졌다"고 지적해서 실제 훅 1개가 추가됨.
• **MODE 1+2 무중단 진행** — 중간에 섹션별 재확인 없이 끝까지 달림 (대표님 원칙 준수).

---

🔧 개선 가능한 부분:

• **queue 재시도 상한 미정** — 재시도 몇 번까지? 무한루프 방지 카운터 필요. → 다음 세션에서 `max_retries: 5` 같은 안전장치 추가 검토.
• **drift 감지 주기 튜닝 필요** — 매 세션 시작마다 전체 스캔은 비용. 변경된 파일만 스캔하는 incremental 방식 고려.
• **handoff 파일 용량 증가 대비 없음** — 8차 넘어가면 서재가 꽉 참. archive 훅(C)에 "90일 이상 → cold storage" 규칙 추가 여지.

---

💡 대표님이 응용할 수 있는 곳:

• **해밀시아 관리대장 시스템**
  → 건물별 일지(원본) / 전체 건물 현황판(인덱스) / Notion DB(백업) 3계층으로 분리.
  → 일지 기록 성공 = 작업 완료. 현황판은 자동 집계 스크립트에 맡김.

• **아이리스 공실 추적**
  → 공실 발생 이벤트(원본 = Notion 페이지) / 대시보드 카운터(인덱스) / 슬랙 알림(백업) 3중화.
  → 이미 유사 구조라 이번 학습이 그대로 이식 가능.

• **브리핑 봇 피드백 학습**
  → 👍/👎 로그(원본 = DB row) / 최근 감정지표(인덱스 = 메모리 캐시) / 주간 리포트(백업 = 파일).
  → queue 재시도 패턴이 Slack API 레이트 리밋 회피에 쓸모 있음.

---

🧠 이어서 생각해볼 것:

1. **"원본은 단 하나여야 하는가?"** — handoff 외에 Notion도 원본 역할을 일부 겸하는 하이브리드가 가능할까? (예: Notion에서만 편집한 내용은 Notion이 원본이 됨)
2. **drift 감지 후 자동 복구 vs 사용자 확인** — 자동으로 handoff→MEMORY 덮어쓰기를 할지, 대표님 확인 받을지 정책 분기 필요.
3. **queue가 계속 쌓이면?** — 실패가 지속되는 건 근본 원인이 있음. queue 길이 임계치 넘으면 알람 발송 검토.
4. **이 패턴을 스킬로 추출 가능?** — "원본 우선 쓰기 + 인덱스 best-effort + queue 재시도"는 범용 패턴. skill-manager로 등록하면 다른 프로젝트에서 재사용 가능.
