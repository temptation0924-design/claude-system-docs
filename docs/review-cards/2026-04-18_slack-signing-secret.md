---

## 2. 큰 그림 비유 — 아파트 3자 구조

| 현실 | 해밀봇 시스템 |
|------|--------------|
| 입주민 (해밀시아) | Slack 사용자·자동발송 이벤트 |
| 경비실 (24시간 근무) | 해밀봇 main.py |
| 관리사무소장 (인증 총괄) | Slack 본사 서버 |

- 관리소장과 경비실장은 **공유 비밀코드**(signing secret)를 미리 정해둠.
- 입주민이 뭔가 요청하면, 관리소장이 **"이건 진짜 우리 입주민 요청"이라는 도장**을 찍어 경비실로 보냄.
- 경비실은 같은 비밀코드로 도장을 재계산해서 **일치해야만 문을 열어줌**.
- 도장이 다르면? → 해커 사칭 가능성 → 전부 차단 ([VERIFY] 실패).

---

## 3. 기술 개념 3가지

### (1) HMAC-SHA256 서명
- **공식**: `도장 = SHA256(비밀키 + 타임스탬프 + 메시지본문)`
- Slack과 해밀봇이 **같은 비밀키**로 계산하니까, 결과 해시가 일치해야 정상.
- 비밀키를 모르는 해커는 절대 같은 도장을 못 만듦. (경비실 도장 위조 불가)

### (2) 전역 싱글 엔트리포인트 설계 (main.py:37)
- 해밀봇 코드 전체에서 `SLACK_SIGNING_SECRET`을 환경변수에서 읽는 곳은 **오직 main.py 37번째 줄 하나**.
- 나머지 5개 플러그인(플래너·일정봇·경비봇·복습봇·회고봇)은 main.py가 전달해주는 값만 참조.
- **아파트로 치면**: 마스터키 원본은 관리사무소 금고에만 있고, 각 동 경비는 관리소가 건네준 임시 복제본만 씀.

### (3) 재시작이 필요한 이유
- Python 프로세스가 시작될 때 `os.environ` → 전역 변수에 **딱 한 번** 로드.
- 이후엔 그 변수가 메모리에 박혀 있음 → Railway에서 값을 바꿔도 **이미 돌고 있는 프로세스는 옛날 값** 그대로.
- **재배포(재시작)**해야 새 비밀코드가 메모리에 올라옴.

## 4. 부동산 도메인 비유 — 마스터키 시스템

- 해밀시아 관리실에 **마스터키 원본 1개**. 각 세대 출입은 마스터키 없이는 불가.
- 마스터키를 분실/교체하면 → **전 세대 현관 도어락 재등록 필수**. 하나만 빼먹어도 그 세대는 먹통.
- 오늘 장애: Slack이 마스터키를 새로 발급했는데(Regenerate), Railway(경비실 사물함)에 **옛날 키를 그대로 둠** → 해밀봇이 새 도장을 못 알아봄 → 5개 플러그인 전원 입실 거부.

## 6. 왜 main.py:37이 "유일한 읽기 지점"이어야 하나

| 이점 | 설명 | 부동산 비유 |
|------|------|-----------|
| **단일 책임** | 읽는 곳이 1곳 → 버그나면 여기만 보면 됨 | 마스터키는 관리소장 1명만 관리 |
| **비밀 최소 노출** | 코드 여기저기서 환경변수 참조하면 실수로 로그에 찍힐 위험 ↑ | 경비마다 마스터키 복제본 주면 분실 위험 ↑ |
| **값 교체 편의성** | 키 회전 시 main.py 한 줄만 신경 쓰면 끝 | 금고 하나만 바꾸면 전체 반영 |

## 7. 대표님 기억 교훈 3가지

1. **로컬 수정 ≠ 배포**: 반드시 `git add → commit → push` 3단계. 로컬만 고치면 프로덕션은 어제 코드.
2. **환경변수는 프로세스 시작 시 1회 로드**: Railway에서 값만 바꾸고 재배포 안 하면 적용 안 됨. **Redeploy 필수**.
3. **Signing Secret 회전 = 양쪽 동기화**: Slack Regenerate 하면 무조건 **Railway 환경변수도 즉시 교체 + 재배포**. 미싱크 시 전체 봇 먹통.
