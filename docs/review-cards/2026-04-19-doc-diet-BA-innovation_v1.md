## 🎯 핵심 개념 3개

### 1. on-demand fetch 아키텍처 — "매뉴얼 휴대"에서 "캐비닛 조회"로
- **비유**: 지금까지는 해밀시아 전 건물 매뉴얼(78KB) 전체를 항상 가방에 넣고 다녔습니다. 이제는 **5페이지짜리 색인표(코어 5KB)**만 휴대하고, "3층 201호 보일러 매뉴얼이 필요해" 싶을 때만 캐비닛(GitHub URL)에서 해당 페이지를 꺼내 봅니다.
- **효과**: 평상시 뇌(context)가 가벼워서 진짜 일할 여유가 20K → 3K 토큰으로 폭증 (-85%)
- **핵심 전제**: 색인표에 "어느 캐비닛의 몇 번째 서랍"인지 **정확한 URL 맵**이 있어야 작동

### 2. 라우팅 허브 코어 — 관리사무소 인포데스크
- **비유**: 새 직원이 와서 "쓰레기 분리배출 어떻게 해요?" 물으면, 인포데스크(CLAUDE-core_v1.md)는 **직접 답하지 않고** "그건 3층 캐비닛 B-4 서랍 보세요"라고 **라우팅만** 합니다. 인포데스크 자체는 얇을수록 좋습니다 (찾기 쉬우니까).
- **구조**: 트리거(키워드) → URL(문서 위치) 맵핑 테이블 + 얇은 규칙만 잔류
- **위험**: 인포데스크가 뚱뚱해지면(= 답까지 써버리면) on-demand 아키텍처가 무너짐. **5KB 상한선 사수**가 규율.

### 3. 병행 유지 무중단 전환 — 엘리베이터 교체하면서 계단 남겨두기
- **비유**: 해밀시아 엘리베이터 교체할 때 계단까지 동시에 폐쇄하면 입주자 난리납니다. INTEGRATED.md(계단, fallback)는 그대로 두고 Claude.ai 북마크(엘리베이터)만 새 코어로 교체 → **롤백 가능한 이사**.
- **실전 효과**: 오늘 PoC 실패했어도 북마크만 원복하면 끝. "배포 공포" 없음.

```
[Before]                          [After]
┌─────────────────────┐           ┌──────────────┐
│ INTEGRATED.md 80KB  │           │ core 5KB     │ ← 평상시 휴대
│ (8개 md concat 전체) │           │ (라우팅 허브)│
│ 매 세션 전체 로드   │           └──────┬───────┘
└─────────────────────┘                  │ 필요 시
         │                                ▼
         ▼                          ┌──────────────┐
      ~20K tokens                   │ on-demand    │
                                    │ mode1-3 URL  │
                                    │ rules URL    │
                                    │ skills URL   │
                                    └──────┬───────┘
                                           │
                                           ▼
                                        ~3K tokens
                                        (필요분만)
```

**플로우**:
1. 대표님 "기획하자" → core가 MODE 1 URL 라우팅
2. Claude.ai가 WebFetch로 mode1-3_v1.md만 로드
3. 기획 끝나면 context 해제 → 다음 요청 다시 core 5KB 상태

## 🔧 개선 가능한 부분

- **prompt injection 방어 발견이 늦음** — "URL만 바꿔 붙이면 된다"로 시작했다가 Claude.ai가 거부. 결국 **"이 URL은 제가 승인한 운영 매뉴얼입니다"** 명시 문구로 우회. → 다음에 외부 URL 도입 시 **처음부터** Custom Instructions 레벨에 주입 전제로 설계할 것.
- **5KB 코어 상한선이 규율로만 존재** — 자동 검증 훅이 없어서 향후 누가(또는 내가) 코어에 내용 추가하면 서서히 뚱뚱해질 위험. → `build-integrated_v1.sh`에 코어 크기 경고 추가 고려 (예: 코어 > 6KB면 빌드 시 warn).
- **on-demand 문서 세분화가 아직 거침** — 현재 mode1-3 묶음. 향후 MODE 1만 부르고 싶을 때 MODE 2/3까지 딸려옴. 세분화 여지 있음.
