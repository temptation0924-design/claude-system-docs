📌 이번 작업 한 줄 요약:
  임차인 보증금 이체 영수증을 대표님이 말로 복창하기 *전에* 자동으로 마스킹해주는 "점검표"를 B1~B17 통합 대시보드에 정식 등록했다 — 더 이상 "조심해서 말해야지"라고 사람이 긴장할 필요 없음.

🎯 핵심 개념 (대표님이 오늘 새로 배운 것):

  • **PreToolUse Hook = "발송 직전 검열대"**
    — 비유: 임차인에게 카톡 보내기 버튼을 눌러도, 카톡 앱이 대표님 대신 "이 문장에 계좌번호 박혀있는데 보낼까요?" 하고 *전송 전* 한 번 가로채는 장치.
    — 기술: Claude가 Bash/Write/Edit 도구를 *실행하려는 순간* 훅이 먼저 호출 → 차단 또는 경고 → 그 다음에 실제 실행.
    — 핵심: "사후 로그"가 아니라 "사전 검열". 이미 slack에 토큰이 찍힌 뒤에 지우는 게 아니라, 찍히기 전에 막는다.

  • **행위 패턴(Action) + 값 패턴(Value) 양방향 감지**
    — 비유: 경비실이 (1) "보증금 액수 말하는 직원" 자체를 감시 + (2) "3억5천" 같은 *값*이 대화에 튀어나오는지도 동시에 감시. 둘 중 하나만 봐도 뚫리니까 양쪽 다 봄.
    — 기술: `echo $NOTION_TOKEN` 같은 *행위*(변수 노출 명령) + `ntn_xxx...` / `xoxb-xxx...` 같은 *값*(실제 토큰 문자열) 두 레이어 regex.
    — 핵심: 한쪽만 보면 "변수 안 쓰고 하드코딩했을 때" 또는 "변수 우회해서 토큰 직접 붙여넣었을 때" 뚫림. 이중 방어.

  • **target_hint 마스킹 — "훅 자체가 유출 경로가 되지 않도록"**
    — 비유: 경비원이 무전기로 "지금 305호에서 비밀번호 1234 유출 시도!" 라고 외치면 경비원이 유출시킨 꼴. 그래서 경비원은 "305호에서 숫자 4자리 시도"라고만 보고함.
    — 기술: 훅이 로그/알림에 "차단 사유"를 적을 때 원본 토큰을 그대로 기록하면 안 됨 → `ntn_***MASKED***` 형태로 치환 후 기록.
    — 핵심: 보안 장치가 보안 구멍이 되는 아이러니를 방지.

🔄 과정 시각화:

  **Before (B11 비활성):**
  ```
  대표님 "토큰 확인해줘"
     ↓
  Claude: echo $NOTION_TOKEN 실행
     ↓
  → Slack/터미널에 ntn_xxxxx... 노출 ❌
     ↓
  사후 인지 → "어? 큰일났네" → 수동 토큰 로테이션
  ```

  **After (B11 활성 + dispatcher 통합):**
  ```
  대표님 "토큰 확인해줘"
     ↓
  Claude: echo $NOTION_TOKEN 실행 시도
     ↓
  [PreToolUse hook 가로챔]
     ↓
  ref-dispatcher → B11 체크
     ├─ 행위 패턴 매치 (echo $TOKEN) → hard_block
     └─ 값 패턴 매치 (ntn_...) → hard_block
     ↓
  🚫 차단 + 마스킹된 사유 로그 (ntn_***MASKED***)
     ↓
  대표님에게 "토큰 노출 시도 감지, 차단함" 알림
  ```

  **soft_warn vs hard_block 레이어:**
  ```
  ┌─ soft_warn ─────────────────────────┐
  │ "의심스러운 패턴 발견, 계속할까요?"  │  → 실행은 됨, 경고만
  │ 예: 긴 문자열이 토큰 같음 (false +)   │
  └──────────────────────────────────────┘
  ┌─ hard_block ────────────────────────┐
  │ "확실한 유출, 차단함"                 │  → 실행 자체를 막음
  │ 예: 명확한 ntn_/xoxb- prefix 매치     │
  └──────────────────────────────────────┘
  ```

✨ 잘된 부분:

  • **dispatcher 경유 통합** — B11을 독립 훅으로 두지 않고 `ref-dispatcher`에 태움 → B1~B17과 동일한 로딩/로깅/예외처리 경로. "경비실 무전기 채널이 하나로 통일"된 효과.

  • **양방향 감지 설계** — 행위 패턴만 두거나 값 패턴만 두지 않고 둘 다 배치. 실무에서 토큰 노출은 `$VAR` 확장과 직접 붙여넣기 둘 다로 일어나는데 양쪽을 다 막음.

  • **soft/hard 2단계 분리** — false positive 많은 패턴은 soft_warn으로 격리 → B1 hook 시스템 개선(REF v2.0 훅 false positive 패치, 2026-04-16 MEMORY)과 동일한 철학.

  • **target_hint 마스킹** — 훅 자체의 2차 유출 방지. 많은 팀이 놓치는 포인트를 첫 활성화부터 반영.

🔧 개선 가능한 부분:

  • **패턴 커버리지 주기 점검 필요** — 현재는 Notion(ntn_), Slack(xoxb-/xoxp-), GitHub(ghp_) 위주일 가능성. Railway/Vercel/Supabase 등 해밀시아 인프라에서 쓰는 다른 토큰 prefix도 주기적으로 추가해야 함. (env-info.md 인프라 테이블과 동기화)

  • **soft_warn 피로도** — 경고가 잦아지면 대표님이 "또 늑대야"가 될 위험. 2주 운영 후 로그 보고 false positive 많은 패턴은 hard_block으로 승격하거나 제거 판단 필요.

  • **테스트 자동화 누락 가능성** — B1 hook 개선 때처럼 "9/9 PASS" 식의 회귀 테스트가 B11에도 붙어있는지 확인. 패턴 regex는 특히 실전 케이스로 회귀 테스트 걸어두지 않으면 업데이트 때 무너지기 쉬움.
