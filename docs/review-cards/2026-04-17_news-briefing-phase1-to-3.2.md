## 🧩 카드 1 — Collector Interface 패턴 (OCP)

**핵심**: 새 주제 추가 시 기존 시스템 무수정, 수집기 클래스 1개만 추가.

**부동산 비유**: 5층 상가 건물에서 4층에 미용실 추가 → 1~3층 기존 세입자 계약서 안 바꿈. "4층 담당자 1명"만 새로 고용. 건물 구조(인터페이스)는 고정, 세입자(수집기)는 무한 확장.

**Before**: 한 파일에 모든 수집 로직 (100줄씩 × 3주제)
**After**: `collectors/` 폴더에 주제별 파일 1개씩. `base.py`가 공통 규격.

**대표님 응용**: 체크인 자동화 (새 OTA 추가 = 어댑터 1개), 청소 업체 관리 (업체 = 프로필 1개)

## 🧩 카드 2 — Claude Web Search vs RSS

**핵심**: 정보 수집 방식 2가지.

**부동산 비유**:
- **RSS = 신문 구독**: 신문사가 "이거 읽어"라고 뿌려줌. 무료·안정적이지만 내가 원하는 것만은 안 옴.
- **Web Search = 전담 비서**: 매일 "영종도 단기임대 동향" 검색 지시. 맞춤형이지만 비용·품질 편차.

**오늘 설계**: 두 방식을 `Collector` 하위 종류로 통합. 안정 소스는 RSS, 맞춤형은 Web Search.

**대표님 응용**: 경쟁사 모니터링 → Web Search, 정부 공지 → RSS

## 🧩 카드 3 — 정합성 게이트 (Validator Pipeline)

**핵심**: 발송 전 "수집 → 필터 → 검증 → 요약 → 발송" 5단계 게이트.

**부동산 비유 (임차인 심사와 동일)**:
1. 수집 = 지원서 접수 (10명)
2. 필터 = 1차 서류 (10→6명)
3. 검증 = 신용조사 + 소득증빙 (6→4명)
4. 요약 = 임차인 프로필 보고서
5. 발송 = 대표님께 최종 후보

**핵심 설계 원칙**: 각 단계 독립 → 한 단계만 교체·개선 가능.

**대표님 응용**: 예약 문의(접수→날짜→보증금→계약), 리뷰 관리(신규→악성필터→응대필요→답변)

## 🧩 카드 4 — Fail-open vs Fail-closed

**핵심**: 검증 시스템 장애 시 "통과시킬까(open) 막을까(closed)"

**부동산 비유**:
- **Fail-closed (엄격)**: 신용조사 회사 전산 장애 → "조사 못 했으니 입주 불가" → 공실 위험
- **Fail-open (관대)**: 같은 상황 → "서류 문제없으니 일단 입주" → 가끔 문제 세입자

**오늘 선택**: Fail-open + 헤더에 "⚠️ 검증 실패 — 원본 전달" 딱지
이유: "잡음 브리핑 > 브리핑 없음"

**대표님 응용**:
- 자동 결제 → **Fail-closed** (돈 걸린 건 막자)
- 브리핑/알림 → **Fail-open** (대체라도 전달)
- 출입 시스템 → 보안 vs 편의 판단

## 🧩 카드 5 — 동적 관심사 관리 (Notion DB + Slack)

**핵심**: 관심사·키워드를 코드에 박지 말고 외부 저장소로 빼기.

**부동산 비유**:
- **Before (하드코딩) = 본사 정책 문서 수정**: 1건 변경에 하루, 담당자 호출 필요
- **After (Notion + Slack) = 사무실 게시판 메모**: 1분, 대표님 직접

**구조**:
```
Notion DB "브리핑 관심사" (키워드·우선순위·활성여부)
    ↑
Slack 명령어 (`/briefing add ...`)
    ↓
다음 브리핑부터 즉시 반영
```

**대표님 응용**:
- 객실 요금표 → Notion으로 (성수기 즉시 반영)
- 자동 응대 FAQ → Notion 템플릿 (시즌별 교체)
- 청소 체크리스트 → Notion ON/OFF
