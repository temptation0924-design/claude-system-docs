🎯 핵심 개념 (오늘 새로 배운 것)

  • **memory_patcher.py의 5개 버그 패턴** — 해밀시아 관리사무소에 비유하면, 임대 카드 정리 담당 직원이 (1) 같은 카드를 중복으로 계속 꽂고 있었고(dedup 없음), (2) 파일함에 카드를 못 꽂아도 아무에게도 보고를 안 했고(silent fail), (3) "만기 7일 지난 카드는 버린다"고 매뉴얼엔 적혀 있는데 실제로는 버리는 동작을 안 하고 있었음(주석만 있고 로직 없음). regex 함정의 본질: **"매치 실패 = 조용한 실패"**. 반드시 `if not match: raise` 가드를 걸어야 함.

  • **rules.md A9 신설 — MEMORY.md 80줄 한도** — 책상 위에 올려둘 수 있는 임대 카드는 최대 80장. 이전 세션에서 "60장"이라고 잘못 인용됐던 걸 오늘 검증해서 정정함. **80줄의 근거**: 시스템 truncation이 200줄에서 잘림 → 40% 안전 마진 확보. 그 이상은 캐비닛(개별 handoff 파일)으로 밀어넣는 구조. 한도를 명문화하지 않으면 "얼마나 많으면 너무 많은가?"가 매번 모호해짐.

  • **노션기록관 큐 재시도 헛짚음 — 표면 에러 → 환경 원인 단정의 위험성** — "임차인 만기 통보가 404로 실패" → "아, Claude Code 환경에서 MCP 접근 불가구나" 로 단정한 것은, 실제로는 **큐 파일에 적힌 DB ID 자체가 오타**였던 상황. 부동산 비유로 옮기면: 세입자에게 등기 보냈는데 반송 → "우체국 시스템이 죽었나 보다" 로 결론 내고 건물 전체 폐쇄 회의 소집한 격. **진짜 원인은 주소 오기재 한 줄**.

✨ 잘된 부분

  • **5개 버그를 한 세션에 모두 잡음** — dedup / silent fail / 7일 롤링 / 빈 desc / 쓰기 실패 패턴을 개별 isolated test로 분리해서 잡은 것. 임대점검으로 치면 "건물 5개 문제를 각 호실별로 분리해서 점검한" 방식.
  • **잘못 기억된 규칙(A2 60줄)을 실제로 검증하고 정정** — 머릿속 기억에 의존하지 않고 rules.md 원본을 직접 확인 → "없는 규칙이었다"는 걸 찾아서 A9로 신설한 것. 대표님 원칙(`no_case_letter_naming` 같은 검증 문화)과 일치.
  • **truncation 200줄의 40% 안전 마진 계산** — 한도를 숫자로 정할 때 근거를 계산식으로 남김. "80줄은 내 느낌"이 아니라 "200 × 0.4 = 80"이라는 규칙 있는 판단.

🔧 개선 가능한 부분

  • **노션기록관 큐 재시도 헛짚음 — 에러 계층 구분 훈련 부족** — "404"라는 표면 증상을 보자마자 "환경 문제"로 점프한 것. 부동산 비유: 세입자 민원 들어왔을 때 1) 세입자 문제인지 2) 호실 문제인지 3) 건물 문제인지 4) 시스템 문제인지 계층별로 좁혀가야 하는데, 바로 4번으로 직행한 격. **교훈: 에러 메시지의 원천(stack trace, 입력 데이터)을 먼저 보고, 환경 문제 단정은 최후 가설로**.

  • **복습 카드 생성 시점의 자동성 검증 필요** — 오늘처럼 "시스템 설정 변경 + 시스템 파일 수정"이 B12 트리거 조건을 충족했음에도, 대표님이 수동으로 "복습해줘" 라고 호출해야 했는지 여부. task-routine.md 자동 트리거 로직이 실제로 발화했는지 다음 세션에서 점검 대상.

💡 대표님이 응용할 수 있는 곳

  • **해밀시아 임대점검 시스템의 "silent fail" 방지** — 지금 memory_patcher에서 잡은 "regex 실패 시 조용히 지나감" 패턴은 임대점검 규칙 1~5에서도 동일하게 발생 가능. 예: "연장합의서 날짜 파싱 실패 → 그냥 넘어감 → 만기일 틀린 채로 보고서 생성". **모든 데이터 파싱 구간에 `if not match: raise` 가드 적용 검토**.

  • **"80줄 한도" 패턴을 다른 문서에도 적용** — MEMORY.md 뿐 아니라 CLAUDE.md, session.md, rules.md도 truncation 위험 있음. 각 파일에 "몇 줄을 넘으면 하위 파일로 분기하라"는 한도 명문화 가능. 이건 해밀시아 운영 매뉴얼을 호실별로 분권화하는 것과 같은 철학.

  • **오진 방지 체크리스트** — 에러 발생 시 (1) 입력 데이터 확인 → (2) 로직 확인 → (3) 환경 확인 순서로만 가설 세우기. 이걸 rules.md에 추가 규칙으로 넣어두면 다음 세션에서도 자동 적용 가능.

🧠 이어서 생각해볼 것

  • **memory_patcher.py에 "규칙 위반" 자동 감지 로직을 넣을 수 있을까?** — 지금은 MEMORY.md에 80줄을 넘으면 사람이 알아채야 함. 매 patch 시점에 `line_count > 80 → 자동 archive`를 넣으면 한도 관리가 완전 자동화됨. 해밀시아로 치면 카드 81장째부터 자동으로 캐비닛 이동하는 시스템.

  • **"노션기록관 큐 DB ID 오타"는 어떻게 재발 방지?** — 큐 파일에 DB ID를 매번 적는 방식 자체가 위험. env-info.md의 DB ID를 환경변수로 참조하는 구조로 바꾸면 오타 불가능해짐. 건물 주소를 매 계약서마다 손으로 쓰지 말고 코드로 참조하는 것과 동일.

  • **B12 자동 트리거가 실제로 발화했는지 로그 확인** — task-routine.md 자동 조건이 걸렸다면 봇이 #claude-study로 카드를 자동 전송했어야 함. 다음 세션에서 `#claude-study` 채널 확인 필요.
