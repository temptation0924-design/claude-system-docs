## 🎯 핵심 개념 (부동산/해밀시아 비유)

### 1. `name: 영문-kebab-case` = **사원증의 사번(社番)**

해밀시아 건물에 신규 용역팀 5명을 채용했다고 생각해 보세요.

- ❌ 이전 방식: 사원증에 "청소아주머니", "경비아저씨"처럼 **한글 별명**만 적음
- ✅ 표준 방식: 사번은 **`cleaning-lead`, `security-chief`** 같은 영문 코드로, 별명은 별도 항목(description)에 기재

**왜?** 건물 관리 시스템(Claude 시스템)이 출입 기록을 남길 때 **한글은 인식 못하는 바코드 리더기**라서, 영문 사번이 없으면 "그 사람 누구세요?" 하며 출입을 거부합니다. 이게 3차 세션 때 발생한 "서브에이전트 권한 거부"의 진짜 원인이었어요.

### 2. `tools: [Read, Write, ...]` = **업무 권한증 (열쇠 꾸러미)**

신규 용역팀원에게 건물 마스터키를 그냥 주지 않죠. **청소팀에게는 각 호실 키**, **경비팀에게는 로비·주차장 키**, 이렇게 필요한 키만 골라서 지급합니다.

- ❌ 이전: "tools" 항목 누락 → 용역팀이 아무 방도 못 들어감 (도구 호출 실패)
- ✅ 표준: 역할별 필요한 도구만 명시 (예: `notion-writer` → Notion 도구만, `janitor` → Bash+파일 도구만)

### 3. `defaultMode: "bypassPermissions"` = **용역팀 자동 출입 승인**

매일 청소하러 오는 아주머니한테 매번 "들어와도 돼요?" 물어보면 업무가 안 되죠. 건물주(대표님)가 **"계약된 용역팀은 매번 묻지 말고 그냥 들여보내"** 하고 경비실에 미리 지시해둔 게 `bypassPermissions`입니다.

- **위치**: `~/.claude/settings.json` (경비실 규정집)
- **효과**: 서브에이전트가 tools 사용할 때마다 "허락할까요?" 팝업 안 뜸

### 4. `model: haiku/sonnet/opus` = **용역팀원의 숙련도 등급**

- **haiku** = 단순 반복 업무 숙련공 (청소, 쓰레기 배출 = janitor)
- **sonnet** = 중간 관리자급 (일정표 작성, 인수인계 = handoff-scribe)
- **opus** = 베테랑 전문가 (학습 코치, 규칙 감사 = study-coach, rule-watcher)

업무 난이도에 맞게 등급을 매겨야 **인건비(토큰비용)도 아끼고 품질도 확보**됩니다.

## 🔄 과정 시각화

```
[3차 세션: 권한 거부 발생]
        │
        ▼
[진단] ─────────────┐
   │                │
   ├─ 원인 1: name이 한글 → 시스템이 subagent_type 인식 실패
   └─ 원인 2: tools 누락 → 도구 호출 자체가 차단
        │
        ▼
[Phase 1: notion-writer 파일럿]
   │
   ├─ name: notion-writer (영문 kebab-case)
   ├─ description: "노션기록관 — Notion DB 기록 전담"
   ├─ tools: [mcp__notion__*]
   ├─ model: sonnet
   └─ ✅ 작동 확인
        │
        ▼
[Phase 2: 5명 일괄 등록]
   │
   ├─ handoff-scribe (인수인계 작성 — sonnet)
   ├─ rule-watcher (규칙 감사 — opus)
   ├─ memory-keeper (메모리 관리 — sonnet)
   ├─ study-coach (복습 카드 — opus) ← 지금 이 카드 작성 중!
   └─ janitor (파일 정리 — haiku)
        │
        ▼
[settings.json 수정]
   └─ defaultMode: "bypassPermissions" → 출입 자동 승인
        │
        ▼
[커밋: d594adb feat: register 5 core agents]
```

## ✨ 잘된 부분

1. **파일럿 먼저 → 일괄 적용** 순서 지킴 (notion-writer 1명 검증 후 5명 일괄)
2. **근본 원인 2개를 모두 진단** (name 문제만 고치고 tools 누락을 놓쳤으면 다시 실패했을 것)
3. **모델 등급을 업무 난이도에 맞춰 차등 배분** — haiku(janitor) ~ opus(study-coach, rule-watcher)
4. **settings.json 추적 대상에 포함** (커밋 686370c) → 향후 롤백 가능

## 🔧 개선점

1. **frontmatter 표준 템플릿이 문서화 안 됨** → `docs/rules/`에 `agent-frontmatter-standard_v1.md` 만들어 두면 다음 에이전트 추가 시 실수 방지
2. **한글 name 금지 규칙이 rules.md에 없음** → B 시리즈(자주 실수 패턴)에 "B19: subagent name은 영문 kebab-case 강제" 추가 권장
3. **bypassPermissions 보안 리스크 미평가** — 실수로 위험 도구(Bash rm 등)를 가진 에이전트가 자동 실행되면? → 에이전트별 tools 화이트리스트 검증 루틴 필요

## 💡 응용처 (부동산/운영 관점)

1. **신규 용역팀 채용 매뉴얼화**: 앞으로 에이전트 추가 시 "사번(name) → 별명(description) → 권한(tools) → 등급(model)" 4단계 체크리스트로 체화
2. **임대점검 봇 확장 시**: rental-inspector에 하위 에이전트 추가할 때 동일 표준 적용 (예: `rental-classifier`, `rental-notifier`)
3. **Cowork 매크로 작업**: Cowork로 자동화할 업무도 "이름/역할/권한/숙련도" 4종 세트로 정의해두면 재사용 쉬움

---

## 🧠 이어서 생각할 것

1. **에이전트 간 권한 충돌 시나리오**: janitor(파일 삭제 권한)와 memory-keeper(파일 보존 역할)가 동시 실행되면? → 우선순위 정책 필요
2. **bypassPermissions의 안전망**: 위험 명령어(rm -rf, git push -f 등) 블랙리스트를 hook으로 강제하는 것이 맞지 않을까?
3. **에이전트 성과 측정**: 각 에이전트가 실제로 몇 번 호출됐고, 몇 번 실패했는지 로깅 → 등급(model) 재조정 근거
4. **에이전트 수가 5명 → 10명 → 20명으로 늘어날 때** 관리 방식: 현재는 개별 .md 파일, 나중엔 레지스트리 DB 필요할 수도
