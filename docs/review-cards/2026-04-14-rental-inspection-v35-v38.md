## 🎓 복습 카드 1 — 특별업무수행 유령링크 예외 (v3.5)

📌 **한 줄 요약**
임차인마스터 Name에 "특별업무수행"이 포함되면, 공실검증.확인=true여도 유령링크 오류 아님.

🎯 **핵심 개념 (부동산 비유)**
  • **특별업무수행 건** = "건물주가 직접 쓰는 방" — 임대차 계약이 없으니 공실 판정 규칙을 적용하면 안 됨
  • **유령링크** = "존재하지 않는 세입자 이름이 달린 열쇠고리" — 정상 판정이면 치워야 하는데, 특별업무수행은 원래 그 자리에 있는 게 맞음

🔄 **판정 플로우**
```
임차인마스터 Row → Name에 "특별업무수행" 포함? 
  YES → 유령링크 규칙 SKIP (정상)
  NO  → 공실검증.확인 = true → 유령링크 오류
```

✨ **잘된 부분**
  • 기존 규칙을 뜯어고치지 않고 **화이트리스트 한 줄**로 예외 처리 (rules.md A3: "돌아가는 규칙 재설계 금지" 준수)

🔧 **주의할 점**
  • Name 매칭이 **정확히 "특별업무수행" 문자열 포함**이어야 함 — "특별업무"만 있으면 매칭 안 됨

💡 **응용 예시**
  • 이번 세션 실제 사례: 공실검증에 확인=true로 남아 있지만 임차인마스터에서 "특별업무수행"으로 등록된 건 → 오류 리스트에서 제외

🧠 **이어서 생각해볼 것**
  "건물주 사용 방" 외에 유사하게 면제해야 할 유형이 또 있는지? (예: 모델하우스, 촬영용 방)

## 🎓 복습 카드 2 — 추가연장X 30일 예외 (v3.6)

📌 **한 줄 요약**
이사예정 rule 7은 **만기일 - 점검일 ≤ 30일**일 때만 오류 발동. 그 이전은 정상.

🎯 **핵심 개념 (부동산 비유)**
  • **추가연장X** = 이 세입자는 더 이상 연장 안 함 → 이사예정 DB에 등록돼야 함
  • **30일 버퍼** = "만기 한 달 전부터 짐 싸기" — 그 이전에 미등록이어도 조급할 이유 없음
  • 점검일이 만기일보다 **31일 이상 먼저**면 아직 이사예정에 안 올라와도 정상

🔄 **판정 플로우**
```
추가연장X = true AND 이사예정 미등록
  ↓ 만기일 - 점검일 계산
  ≤ 30일 → 🔴 오류 (빨리 등록하라)
  > 30일 → 🟢 정상 (아직 여유 있음)
```

✨ **잘된 부분**
  • 단순 "미등록 = 오류"에서 **시간 축을 추가**해 false positive 제거
  • "급하지 않은 건 건드리지 않는다" 원칙 반영

🔧 **주의할 점**
  • 30일 = 달력일 기준(30 days), 근무일 아님
  • 점검일 = 스크립트 실행일(today)

💡 **응용 예시**
  • 이번 세션: 만기 60일 전 추가연장X 건 → 기존 v3.5에선 오류, v3.6에선 정상 처리

🧠 **이어서 생각해볼 것**
  30일이 적정한가? 15일 / 45일도 후보 — 실제 이사 준비 리드타임 데이터로 튜닝 가능

## 🎓 복습 카드 3 — 기간내퇴거 case C 제외 (v3.7)

📌 **한 줄 요약**
아이리스 rule 7은 **퇴거날짜지정만** 대상. 기간내퇴거는 이사예정을 건너뛰는 정상 플로우라서 제외.

🎯 **핵심 개념 (부동산 비유)**
  • **퇴거날짜지정** = "이사 나갈 날짜 예약" → 이사예정 DB에 올려야 cross-check 대상
  • **기간내퇴거** = "계약 만기까지 살다가 그냥 나감" → 이사예정 DB를 **거치지 않는 게 정상 플로우**
  • case C에 기간내퇴거를 포함하면 "정상 플로우를 오류로 보고"하는 false positive 발생

🔄 **판정 플로우 (Before/After)**
```
[v3.6 이전]
아이리스 rule 7: 퇴거날짜지정 OR 기간내퇴거 → 이사예정 미등록 시 오류
  ❌ 기간내퇴거가 오탐됨

[v3.7]
아이리스 rule 7: 퇴거날짜지정만 → 이사예정 미등록 시 오류
  ✅ 기간내퇴거는 별도 플로우로 처리
```

✨ **잘된 부분**
  • 원래 규칙의 **적용 범위를 좁혀** 해결 — 규칙 추가가 아니라 조건 제한
  • rules.md A3 원칙(기존 규칙 뜯어고치지 않기)을 **좁은 면제로** 준수

🔧 **주의할 점**
  • "퇴거" 용어가 2개 상태로 분기되는 점 주의 (퇴거날짜지정 ≠ 기간내퇴거)
  • case A/B/C/D 알파벳 별명은 금지(feedback_no_case_letter_naming_v1) — 판정결과 이름으로 부르기

💡 **응용 예시**
  • 이번 세션 실제 사례: 만기 도달해서 그냥 나간 세입자 → 이사예정 미등록이지만 정상

🧠 **이어서 생각해볼 것**
  기간내퇴거는 어느 DB에서 검증해야 하나? (임차인마스터 status 흐름?)

## 🎓 복습 카드 4 — 삼삼엠투 계약금 예외 (v3.8)

📌 **한 줄 요약**
신규입주자DB 비고에 "삼삼엠투" 포함 시, 계약금 미완납 오류 발동 안 함 (잔금 일시불 모델).

🎯 **핵심 개념 (부동산 비유)**
  • **일반 계약** = "계약금 → 중도금 → 잔금" 3단계 → 계약금 미완납이면 오류
  • **삼삼엠투 계약** = "입주 당일 잔금 일시불" → 계약금 단계 자체가 없음
  • 이걸 일반 규칙으로 체크하면 "당연히 0원인데 왜 0원이냐"는 false positive 발생

🔄 **판정 플로우**
```
신규입주자 Row → 비고 필드에 "삼삼엠투" 포함?
  YES → 계약금 체크 SKIP
  NO  → 계약금 ≥ 지정액 아니면 🔴 오류
```

✨ **잘된 부분**
  • **비즈니스 모델 차이**(플랫폼별 결제 방식)를 데이터 검증에 반영
  • 메모리에 저장된 도메인 지식(feedback_samsam_m2_lease_v1)을 규칙으로 승격

🔧 **주의할 점**
  • "삼삼엠투" 문자열은 **비고 필드**에서 검색 (계약 경로 필드 아님)
  • 삼삼엠투 외에 유사 플랫폼(에어비앤비 장기?)도 같은 처리 필요한지 점검 필요

💡 **응용 예시**
  • 이번 세션 실제 사례: 삼삼엠투 플랫폼 예약 건 → 계약금 0원이지만 잔금 완납 → 정상

🧠 **이어서 생각해볼 것**
  다른 숙박 플랫폼(에어비앤비, 야놀자 등)도 비슷한 정산 구조인지? 화이트리스트 확장 후보
