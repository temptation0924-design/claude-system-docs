## 카드 1 — 대환대출 분석 구조 (부동산 운영 도메인)

**한 줄 요약**:
대환대출은 "건물에 잠겨있던 보증금을 풀어내서, 그 자리를 월세 임차로 채우는 리노베이션"입니다.

**핵심 개념**:
- **대환대출 메커니즘** — 추가대출로 보증금을 토해내고 → 그 호실을 월세로 재임대 → 고정수익(월세) 증가. "전세 5천짜리 호실을 보증금 500/월 70으로 갈아끼우는 것"과 동일
- **수익 성립 조건** — `대출이자 증가분 < 월세 증가분`. 이게 깨지면 "이자 갚으려고 월세 받는" 적자 구조 = 부동산 레버리지 손익분기
- **남양리 2027-10 케이스** — 이자 +119만 / 월세 +342만 → 순수익 +223만/월 = 연 2,676만 추가 현금흐름
- **본질** — "대출"이 아니라 자본구조 재편. 죽은 돈(보증금) → 살아있는 돈(월세) 전환

**과정 시각화**:
```
[Before]                          [After]
보증금 5억(잠김) + 대출 3억   →   보증금 2억 + 대출 6억
월세 100만/월                     월세 442만/월
이자  80만/월                     이자 199만/월
─────────────                    ─────────────
순수익 20만/월                    순수익 243만/월  (+223만)
```

**잘된 부분**:
- 케이스별 숫자를 차액 형태(이자 vs 월세)로 잡아내서 의사결정 직관화
- "성립 조건" 1줄 공식 = 다른 호실에 즉시 적용 가능한 재사용 판단 기준

**개선 가능한 부분**:
- 금리 변동 리스크 미반영 — 변동금리면 +119만이 +180만으로 늘 수 있음. 시나리오 3개(낙관/중립/비관) 동시 산출 권장
- 공실 리스크 미반영 — 월세 342만은 100% 임대 가정. 공실률 10% 적용 시 실제 +189만으로 줄어듦

**응용처**:
- 해밀시아 다른 호실 대환 검토 시 → "이자 vs 월세" 표 1장으로 의사결정 자동화
- 신규 매입 검토 시 → "전세 끼고 매입 vs 대출 끼고 월세 운영" 비교
- 투자자 IR 자료 → "자본구조 재편으로 월 +223만 창출" 스토리

**이어서 생각해볼 것**:
- 해밀시아 전 호실 대환 시뮬레이션을 스킬화할 가치가 있는가? (반복 의사결정이면 YES)
- 대환 후 5년차 누적 현금흐름 vs 매각 시나리오 비교는?

## 카드 2 — haemilsia-property-card 스킬 패턴 (시스템 도메인)

**한 줄 요약**:
"손님한테 보여줄 매물 사진"을 만들 때, 단순 이미지가 아니라 **모델하우스 짓고 → 카메라 위치 잡고 → 셔터 누르는** 3단계 파이프라인.

**핵심 개념**:
- **HTML → HTTP 서버 → 브라우저 캡처** — "도면(HTML) 그리고 → 모델하우스 임시로 세워(서버) → 정해진 위치에서 사진 찍기(브라우저)"
- **자족형(self-contained) HTML** — 이미지를 base64로 본문에 박아넣기 = "전단지에 사진을 풀로 붙이지 않고 인쇄로 박아버리는 것". 외부 파일 의존 0
- **viewport + reload 필수 쌍** — 카메라 화각만 바꾸면 화면이 안 바뀜. 다시 로드해야 변경된 화각으로 렌더. "줌 다이얼만 돌리고 셔터 누르면 이전 사진 나오는 카메라 버그"
- **채팅 업로드 이미지 ≠ 파일** — 톡으로 받은 사진은 휘발성 미리보기. 도면에 붙이려면 "파일로 저장해주세요" 요청 필수

**과정 시각화**:
```
1. HTML 작성 (도면 + base64 이미지 임베드)
       ↓
2. 로컬 HTTP 서버 기동 (모델하우스 임시 오픈)
       ↓
3. browse() 호출 — viewport 설정
       ↓
4. ⚠️ reload() — 빠지면 1280px 기본값으로 찍힘
       ↓
5. PNG 캡처 (최종 사진)
       ↓
6. HTTP 서버 종료 (모델하우스 철거)
```

**잘된 부분**:
- base64 임베드로 단일 HTML 자족 — 결과물 1개 파일로 전달 가능 (전단지 1장처럼)
- 채팅 업로드 한계를 사용자 행동(저장 요청)으로 우회 — 시스템 한계를 워크플로우로 해결

**개선 가능한 부분**:
- viewport → reload 누락 함정 — SKILL.md 상단에 "reload 빠뜨리면 무조건 망함" 경고 박스 필수
- 임시 HTTP 서버 종료 안 하면 포트 점유 → 다음 호출 실패. try/finally 패턴으로 강제 종료 보장 권장

**응용처**:
- 임차인 안내문 자동 생성 (호실 정보 + 사진 + QR코드)
- 월간 운영 리포트 PDF화 (대시보드 HTML → 캡처 → 첨부)
- 대환대출 분석 비교표를 이미지 한 장으로 → IR/투자자 공유용

**이어서 생각해볼 것**:
- viewport+reload 같은 "쌍으로 묶여야 작동하는 패턴"을 스킬에서 강제할 방법? (래퍼 함수 vs 문서 경고)
- property-card를 일반화해서 "HTML→이미지 변환 공통 스킬"로 추출 가능한가?
