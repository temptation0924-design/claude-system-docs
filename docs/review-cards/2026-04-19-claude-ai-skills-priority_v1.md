🏷️ 트리거: 자동 — 시스템 설정 변경(Custom Instructions 재편) + 새 개념 도입(idea-validation 5-Phase) + MODE 1→2 사이클 완료

📌 이번 작업 한 줄 요약:
  Claude.ai라는 **"본사"**의 규정집(Custom Instructions)을 얇게 다듬고, 실무 매뉴얼(Skills)에 업무 절차를 위임해서, 지점(Skills)이 묻히지 않게 정리한 날. 동시에 Claude.ai가 **"회의실 화이트보드"**(시각화) 역할을 한다는 걸 재발견.

🎯 핵심 개념 (대표님이 오늘 새로 배운 것):

  • **Custom Instructions vs Skills 우선순위 충돌**
    — 본사 규정(CI)이 두꺼우면 지점 매뉴얼(Skills)을 덮어쓴다. 해밀시아로 치면 "본사 전체 규정에 임차인 응대 매뉴얼까지 다 적어버리면, 지점장이 가진 상황별 매뉴얼이 먹히지 않는" 상황.
    — 해결: CI는 **라우팅 + 시각화 지침 + 안전지침**만. 워크플로우 실체(MODE 1 7단계 등)는 **Skills로 이관**.

  • **Claude.ai의 진짜 가치 = 시각화 캔버스**
    — Claude Code는 "공사 현장"(파일 만들고 배포), Claude.ai는 "설계 사무소 화이트보드"(표·매트릭스·Phase 상태판).
    — 지금까지 대화형으로만 썼던 건, 임대 사업 계획을 카페에서 말로만 설명한 셈. artifact·diagram으로 **보이게** 해야 사고가 정리됨.

  • **idea-validation 스킬 5-Phase 구조**
    — Phase 1 컨텍스트 파악 → Phase 2 모드 결정(Startup/Builder) → Phase 3 진단 질문(YC 6대, 조건부 스킵) → Phase 4 대안 탐색(α/β/γ 난이도) → Phase 5 Design Doc 고정.
    — 부동산 비유: **매물 접수(Ph1) → 판매/임대 전략 선택(Ph2) → 매물 실사(Ph3) → 가격대별 대안 시뮬(Ph4) → 최종 제안서(Ph5)**.

✨ 잘된 부분:
  • Custom Instructions를 과감히 다이어트 — "본사 규정이 두꺼울수록 지점이 죽는다"는 구조적 통찰
  • Claude.ai의 역할을 "대화"에서 "시각화 캔버스"로 재정의 — 보증금 알림·월세 체크 같은 실무 기획에 즉시 활용 가능
  • idea-validation Phase를 5단계로 명확히 구획 — 각 Phase가 부동산 매물 처리 프로세스와 1:1 매칭

🔧 개선 가능한 부분:
  • **Phase 3 조건부 스킵 로직이 아직 불명확** — 어떤 조건에서 YC 6대 질문을 건너뛸지 명시되지 않으면, 실전에서 "이번엔 스킵인가 아닌가" 혼동 발생 가능. 스킵 조건을 결정 테이블로 명문화 필요.
  • **Claude.ai 시각화 가치**를 재발견했지만, "언제 Code 대신 Claude.ai를 쓸지"의 판단 기준이 아직 암묵지. 도구 라우팅 규칙에 "시각화가 필요하면 Claude.ai" 조항을 명시할 여지.

💡 대표님이 응용할 수 있는 곳:
  • **해밀시아 임대점검 기획**: Claude.ai 표·매트릭스로 건물×호수×점검항목 상태판을 한눈에 시각화 (지금까지는 대화로만 물어봤던 부분)
  • **보증금/월세 알림 설계**: idea-validation 5-Phase를 그대로 적용 — Ph1(현황) → Ph2(Startup이냐 Builder냐) → Ph3(누가·언제·왜) → Ph4(알림 수준 α/β/γ) → Ph5(확정 Design Doc)
  • **본사-지점 구조 비유**: 앞으로 어떤 시스템 설계든 "규정(global)과 매뉴얼(local) 분리" 원칙을 적용. 규정이 두꺼워지면 매뉴얼이 죽는다.
  • **Cowork 지침도 동일 점검 필요**: Cowork용 지침이 두껍다면 Skills/툴 매뉴얼이 묻힐 가능성 — 같은 다이어트 대상인지 확인
