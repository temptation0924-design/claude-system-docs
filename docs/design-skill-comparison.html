Marketing Design Planner Skill 비교 분석

D2-test vs D3-test vs 새 설계안 — 좋은 점은 살리고, 약한 점은 고친다

3개 파일, 한눈에 보기
D2-test

초기 버전

348줄 | 기본 4-엔진 구조
  • 감지 → 질문 → 4엔진 → 패키지
  • 문서/랜딩/블로그 3종 패키지
  • 실행 스킬 연결 (7종)
  • Common Mistakes 가이드
D3-test

GAP 패치 버전

348줄 | D2 + 대표님 직접 추가 규칙
  • D2의 모든 것 +
  • G1 참고자료 적응 (질문 자동 축소)
  • G2 비주얼 엔진 필수 실행
  • G5 원라이너 생략 금지
  • 톤→비주얼 자동 매핑표
  • SNS/카드뉴스 패키지 추가
새 설계안

워크플로우 재설계

~250줄 | 완전히 다른 접근
  • 3경로 분기 (신규/수정/핸드오프)
  • 복잡도 게이트 (Fast/Standard/Extended)
  • 역할별 핸드오프 4종
  • Quality Gate 8항목
  • 도메인 라우팅 (부동산/뷰티 등)
  • Failure Recovery 4패턴
핵심 차이: D2/D3는 "매번 처음부터 기획"하는 일방향 파이프라인. 새 설계안은 "상황에 따라 다른 경로"로 진입하는 라우터.
2
10개 차원 상세 비교
비교 차원 D3 (현재) 새 설계안 승자 이유
워크플로우 분기 없음 — 항상 같은 경로 신규 / 수정 / 핸드오프 3경로 새 설계안 "고쳐줘"에도 처음부터 시작하면 비효율
복잡도 적응 G1 참고자료 축소만 Fast / Standard / Extended 3단계 새 설계안 간단한 건 빠르게, 복잡한 건 깊게
수정 요청 대응 없음 (처음부터 재시작) 진단 → 유지/수정/전환 3분류 새 설계안 실무에서 "수정"이 "신규"보다 더 자주 발생
핸드오프 스킬 연결만 역할별 분리 (기획/디자인/카피/이미지) 새 설계안 협업자에게 바로 전달 가능
품질 검증 없음 Quality Gate 8항목 체크리스트 새 설계안 빠뜨린 게 없는지 자동 점검
메시지 엔진 StoryBrand + 원라이너 + 헤드라인3종 + CTA + Objection planning core만 (간략) D3 카피 품질의 핵심. 구체적 프레임이 필수
비주얼 엔진 톤→컬러/레이아웃/서체 매핑표 + 프롬프트 생성 "visual cues" 한 줄만 D3 실제 디자인 결정에 매핑표가 필수
패키지 템플릿 4종 (문서/랜딩/블로그/SNS) 구체적 마크다운 테이블 구조만 (복붙 불가) D3 템플릿이 있어야 일관된 출력
실행 스킬 연결 /frontend-design, /pptx 등 7종 매핑 없음 D3 해밀시아 생태계와의 연결이 핵심 가치
대표님 G규칙 G1 / G2 / G5 명시 없음 D3 실전 피드백에서 나온 규칙 — 반드시 유지
새 설계안 5승 — 구조/분기/적응력
D3 5승 — 실행 디테일/카피 품질
3
D3 (현재 스킬) 장점/단점

+ 장점 — 유지할 것

  • 4엔진 시스템 — 메시지/구조/콘텐츠/비주얼이 항상 세트로 나옴
  • StoryBrand 원라이너 — 모든 카피의 방향타 역할
  • 톤→비주얼 매핑표 — "신뢰감"만 골라도 컬러+서체+레이아웃이 자동 결정
  • 4종 패키지 템플릿 — 문서/랜딩/블로그/SNS별 구체적 출력 포맷
  • G1/G2/G5 규칙 — 대표님 직접 피드백에서 나온 실전 규칙
  • 실행 스킬 연결 7종 — 기획→실행 자연스러운 핸드오프
  • 질문 형식 규칙 — "한 번에 하나씩, 선택지 중심" 명확

- 단점 — 고칠 것

  • 일방향 파이프라인 — "수정해줘"에도 처음부터 전체 재실행
  • 복잡도 무시 — 간단한 썸네일이나 복잡한 규제 산업이나 같은 흐름
  • 수정 워크플로우 부재 — 기존 초안 개선 요청에 대응 불가
  • 품질 검증 없음 — 패키지 완성 후 빠뜨린 게 있는지 체크 안 함
  • 실패 복구 없음 — 애매한 요청에 대한 탈출 전략 부재
  • 역할별 핸드오프 없음 — 디자이너/카피라이터별 분리 출력 불가
  • 도메인 특화 없음 — 부동산과 뷰티가 같은 방식으로 처리됨
4
새 설계안 장점/단점

+ 장점 — 가져올 것

  • 3경로 워크플로우 — 신규/수정/핸드오프 상황별 최적 경로
  • Complexity Gate — Fast(3~5) / Standard(4~6) / Extended(6~8) 적응
  • Revision Workflow — 진단 → 유지/수정/방향전환 체계적 수정
  • 역할별 핸드오프 — Planner/Designer/Copywriter/Image gen 분리
  • Quality Gate — 8항목 완성도 체크리스트
  • Failure Recovery — 애매/급한/모순/불안정 4가지 탈출 패턴
  • Domain Routing — 도메인별 설득 논리 차이 반영
  • Essentials 체크 — 6가지 필수 요소 중 몇 개가 이미 있는지 판단

- 단점 — 보완할 것

  • 메시지 엔진 약함 — StoryBrand/원라이너 같은 구체적 프레임 없음
  • 비주얼 디테일 없음 — 톤→컬러/서체 매핑이 없어서 추상적
  • 패키지 템플릿 부재 — 출력 포맷이 테이블뿐, 복붙 불가
  • 스킬 연결 없음 — /frontend-design 등 해밀시아 생태계 연결 끊김
  • G규칙 없음 — 대표님 실전 피드백 규칙 미반영
  • 외부 파일 참조 — references/ 폴더 의존 (단일 파일이 더 안전)
  • 한글화 부족 — 영어 위주 서술 → 대표님 확인 시 불편
