# 사전검증 (Preflight Check) 규칙

## 트리거
- **자동 실행**: MODE 1 기획 워크플로우 5번(writing-plans) 완료 후 자동 진입
- **수동 트리거**: "사전검증", "프리플라이트", "preflight", "기획검증해줘"
- **실행 전 확인**: "배포 전 확인", "실행해도 되나", "이거 돌려도 돼?", "이거 적용해도 돼?"
- 90% 미만 FAIL 시 → 자동 수정 → 재검증 반복 (PASS까지)

> "검증해줘", "리뷰해줘" 등은 MODE 1 컨텍스트에서 Preflight로, MODE 3에서는 /qa·/review로 라우팅

## 검증 프로세스
1. Notion 에러로그 DB(a5f92e85220f43c2a7cb506d8c2d47fa) 조회 → 유사 에러 패턴 확인
2. Agent 1 (설계 검증): 로직·구조·변수·의존성 점검
3. Agent 2 (실행 검증): 파일경로·토큰·환경변수·에러패턴 교차확인
4. Agent 3 (엣지케이스): 특수문자·빈입력·타임아웃·stdin충돌·UX 점검
5. **종합 판정 + 점수 표시 필수**:
   - CRITICAL 0건 = PASS / CRITICAL 1건+ = FAIL (수정 필수)
   - **종합 점수(%) 공식**: `100% - (CRITICAL × 15%) - (WARNING × 3%)`
   - **90% 미만 FAIL** → 자동 수정 → 재검증 반복 (PASS까지)
   - **보고 형식 예시**: `검증 결과: PASS (94%) | CRITICAL 0건, WARNING 2건, INFO 3건`
   - ⚠️ 점수 미노출 시 규칙 위반 (과거 위반 #16 재발 방지)

## 심각도 분류
- 🔴 CRITICAL: 실행 시 즉시 에러, 데이터 손실, 보안 위험 → 배포 차단
- 🟡 WARNING: 기능 작동하나 비효율/에러핸들링 부족 → 개선 권장
- 🟢 INFO: 스타일/주석/대안 존재 → 참고
